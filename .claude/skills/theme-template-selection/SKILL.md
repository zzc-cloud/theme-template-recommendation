# 魔数师主题定位与模板推荐（子 Skill）

## 概述

本 Skill 是 `theme-template-recommendation` 的子 Skill，负责从需求澄清阶段（阶段 0）完成后，执行主题定位和模板推荐。

**本 Skill 的职责**：
- **主题定位**：通过统计聚合 + 层级导航两条路径生成候选主题
- **指标裁决**：LLM 裁决判断主题可用性并精筛指标
- **模板推荐**：基于精筛指标检索匹配模板并进行缺口分析

---

## 输入参数

由主 Skill 阶段 0 传递的结构化数据：

```
{
  user_question: "用户原始问题",
  normalized_question: "规范化问题描述",
  analysis_dimensions: [
    {
      dimension: "分析维度名称",
      matched_indicators: [
        { indicator_id: "...", alias: "...", similarity_score: 0.85 }
      ]
    }
  ],
  filter_indicators: [
    { indicator_id: "二级账务机构名称", value: "南京分行" }
  ],
  matched_indicators: ["indicator_id_1", "indicator_id_2", ...]
}
```

## 🔍 探查输出要求

**⚠️ 重要：实时串行输出**，每个步骤执行后立即输出，不得批量输出。

规则：
1. 本 Skill 开始时输出 `[主题定位与模板推荐] 开始执行`
2. 每次工具调用后立即输出调用和结果
3. 本 Skill 完成时输出 `[主题定位与模板推荐] 执行完成`

## 执行流程

### 阶段 1：主题定位与指标裁决

**目标**：从阶段 0 获取确认的分析维度及关联指标，通过两条并行路径生成候选主题，合并后精筛覆盖用户需求的指标。

**流程图（两条并行路径）**：
```
1.1 统计聚合 ──┐
              ├──→ 候选主题合并去重 → 1.3 指标补全 → 1.4 LLM裁决
1.2 层级导航 ──┘
```

**说明**：
- **路径A（1.1 统计聚合）**：从用户确认的分析维度关联的指标，聚合指标归属的主题
- **路径B（1.2 层级导航）**：从"自主分析"根节点出发，基于用户问题 + 分析维度，结合 ontology 中所有板块下的所有主题及其路径，生成候选主题
- 两条路径的候选主题**合并去重**后，共同进入后续指标补全和 LLM 裁决

#### 1.1 THEME 统计聚合

从所有匹配指标中聚合 THEME 信息：
1. 调用 `aggregate_themes_from_indicators(matched_indicators)` 批量获取指标的主题信息
2. **按相似度加权统计各 THEME 频次并排序**
3. **动态选择：所有加权频次 >= 0.6 的 THEME 作为候选主题**（不再固定返回数量）

**⚠️ 相似度加权机制（核心改进）**

不同分析维度的指标相似度差异很大（0.31 ~ 0.98），在聚合主题时需要按相似度加权。

**加权公式**：
```
weighted_frequency = Σ(indicator_similarity_score)
```

| 指标 | 相似度 | 对 Theme A 的贡献 |
|------|--------|------------------|
| 指标1 | 0.92 | 0.92 |
| 指标2 | 0.85 | 0.85 |
| 指标3 | 0.38 | 0.38 |
| 指标4 | 0.92 | 0.92 |
| 指标5 | 0.31 | 0.31 |
| **Theme A 加权和** | - | **3.38** |

**⚠️ 指标去重规则**：
同一指标（相同 `indicator_id`）在不同分析维度中可能重复出现，取**最大相似度**计入加权和。

**输出示例**：
```json
{
  "candidate_themes": [
    {
      "theme_id": "THEME.xxx",
      "theme_alias": "个人存款",
      "theme_level": 4,
      "theme_path": "自主分析 > 负债板块 > 个人存款 > 个人存款",
      "frequency": 5,
      "weighted_frequency": 3.38,
      "matched_indicator_ids": ["INDICATOR.xxx", ...]
    }
  ]
}
```

**排序规则**：按 `weighted_frequency` 降序排序，而非原始 `frequency`。

**动态阈值机制**：
- **阈值**：`weighted_frequency >= 0.6`
- **兜底规则**：若所有主题的加权频次都 < 0.6，返回加权频次最高的主题（确保至少有结果）

> **注意**：
> 1. `aggregate_themes_from_indicators` 返回结果中已包含 `theme_path` 字段
> 2. 加权和计算需要 Skill 执行时传入各指标的相似度分数，由 Skill 内部汇总

#### 1.2 层级路径导航

**目标**：基于用户问题 + 分析维度，结合 ontology 中所有板块下的所有主题及其路径，从"自主分析"根节点出发，批量获取板块下所有主题，筛选候选主题。

**导航流程**：

```
1. 调用 get_sectors_from_root() 获取所有板块
2. 基于用户问题和分析维度，判断哪些板块可能包含相关主题（可多选 4-8 个）
3. 对每个相关板块调用 get_sector_themes(sector_id) 一次性获取所有主题（含完整路径）
4. LLM 在批量结果中基于主题名称 + 路径上下文综合筛选候选主题
```

**导航 Prompt**：

```
你是银行数据分析专家。请基于用户问题和分析维度，通过层级导航工具从"自主分析"根节点出发，生成候选主题列表。

【用户原始问题】
{user_question}

【用户确认的分析维度】
{for dim in analysis_dimensions}
- 「{dim}」
{/for}

【层级导航任务】

第一步：调用 get_sectors_from_root() 获取所有板块。
第二步：结合用户问题和分析维度，判断哪些板块可能包含相关主题（不遗漏：优先选择语义关联度高的板块，可多选 2-4 个）。
第三步：对每个相关板块调用 get_sector_themes(sector_id) 一次性获取该板块下所有主题（无论深度，含完整路径）。
第四步：在每个板块的全部主题中，基于主题名称 + 完整路径上下文筛选1-3个候选主题会汇总输出候选主题列表
- 主题名称是否与用户问题语义匹配
- 主题路径中的中间节点是否提供额外上下文（如「贷款」下 vs 「客户」下）
- 同一板块内的主题数量和分布
- 主题深度（浅层主题可能更通用，深层主题可能更精确）


【输出格式】
候选主题列表（按匹配度降序）：
- theme_id: "..."
  theme_alias: "..."
  theme_path: "..."
  theme_level: 5
  match_reason: "为什么这个主题与用户问题相关"
```

**两条路径汇合**：

1.1 统计聚合和 1.2 层级导航各自生成候选主题后：
- 合并去重（按 theme_id）
- 保留两条路径的全部上下文（指标来源 / 层级来源）
- 共同进入 1.3 指标补全和 1.4 LLM 裁决

#### 1.3 指标补全

对每个推荐主题执行指标补全操作：

**步骤 1.3.1**：调用 `get_theme_filter_indicators(theme_id)` 获取主题下的全量筛选指标

**筛选指标识别规则**（在 MCP 工具内部实现）：
- **时间筛选指标**：指标别名包含 `数据日期` 或 `ETL数据日期`
- **机构筛选指标**：指标别名包含 `机构名称`、`机构编号`、`管理机构名称`、`管理机构编号`、`账务机构名称`、`账务机构编号`

**步骤 1.3.2**：调用 `get_theme_analysis_indicators(theme_id)` 获取主题下的全量分析指标

分析指标为主题下除去筛选指标外的所有指标。

#### 1.4 LLM 裁决

**目标**：基于 1.2 补全的全量指标（筛选指标 + 分析指标），由 LLM 分两步完成裁决：
1. **第一步**：判断当前主题是否能够支撑用户需求
2. **第二步**：在能够支撑的前提下，精筛出覆盖用户需求的具体指标

**两步裁决均在同一次 LLM 调用中完成**，通过输出结构区分。

**裁决 Prompt**：

```
你是银行数据分析专家，需要从主题的全量指标中精准定位能覆盖用户需求的指标。

【用户原始问题】
{user_question}

【用户确认的分析维度】
{for dim in analysis_dimensions}
- 「{dim}」关联指标: {dim.matched_indicators}
{/for}

【当前主题】
主题名称: {theme_name}
主题路径: {theme_path}

【主题下全量筛选指标】
{for ind in filter_indicators}
- {ind.alias}（类型: {ind.type}）
{/for}

【主题下全量分析指标】
{for ind in analysis_indicators}
- {ind.alias}
{/for}

【裁决任务】

第一步：主题可用性判断
判断该主题是否能够支撑用户的分析需求。
判断依据：
- 主题的业务领域是否与用户需求匹配？
- 主题下的分析指标是否覆盖用户的核心分析意图？
- 主题下的筛选指标是否支持用户所需的筛选条件（机构、时间等）？

第二步：指标精筛（仅在主题可用时执行）
从筛选指标和分析指标两部分中，分别定位出能覆盖用户分析需求的指标。

精筛原则：
- 筛选指标：保留与用户筛选条件（机构、时间）匹配的指标
- 分析指标：保留与用户分析维度语义相关、或有补充分析价值的指标
- 排除与用户需求无关的指标
- 指标数量不设上限，以完整覆盖用户需求为准

【输出格式（JSON）】
{
  "theme_id": "...",
  "theme_name": "...",
  "is_supported": true/false,
  "support_reason": "主题可用性判断理由",
  "selected_filter_indicators": [
    {
      "indicator_id": "...",
      "alias": "指标别名",
      "type": "机构筛选指标/时间筛选指标",
      "reason": "选取原因"
    }
  ],
  "selected_analysis_indicators": [
    {
      "indicator_id": "...",
      "alias": "指标别名",
      "reason": "选取原因"
    }
  ],
  "unsupported_dimensions": ["无法覆盖的分析维度（如有）"]
}
```

**裁决结果处理**：

| 裁决结果 | 处理方式 |
|---------|---------|
| `is_supported: true` | 进入阶段 2，以精筛后的指标集合为基准做模板匹配 |
| `is_supported: false` | 降级为备选主题，在输出中标注"不推荐"并说明原因 |
| 所有主题均不支持 | 向用户说明无匹配主题，建议换词重新描述 |

**本阶段最终输出（传递给阶段 2）**：

```
recommended_themes = [
  {
    theme_id: "...",
    theme_name: "...",
    theme_path: "...",
    is_supported: true,
    selected_filter_indicators: [...],   // 精筛后的筛选指标
    selected_analysis_indicators: [...], // 精筛后的分析指标（含语义增强描述）
    unsupported_dimensions: []
  },
  ...
]
```

### 阶段 2：模板推荐

**目标**：仅对阶段 1 裁决后 `is_supported: true` 的主题进行模板推荐，基于精筛后的指标集合检索匹配度达标的模板，并由 LLM 结合用户问题对每个模板的指标构成进行开卷分析，输出可用性说明与缺口提示。

**整体流程**：

```
2.1 MCP 检索模板（带降级策略）
        ↓
   有达标模板（>= 80%）？
        ├─ 是 → 返回达标模板列表
        └─ 否 → 降级推荐：覆盖率最高 + 热度最高的模板（最多 2 个）
        ↓
2.2 LLM 可用性与缺口分析
        ↓
   结合用户问题，分析每个模板的指标构成，输出可用性与缺口说明
```

#### 2.1 MCP 检索模板

对每个 `is_supported: true` 的推荐主题，调用：

```
get_theme_templates_with_coverage(
  theme_id="...",
  matched_indicator_aliases=[
    ...selected_filter_indicators 的 alias,
    ...selected_analysis_indicators 的 alias
  ],
  top_k=10
)
```

**工具过滤逻辑**：

1. **第一步**：过滤掉 `heat = 0` 的模板（无使用记录的模板不参与推荐）
2. **第二步**：在 `heat > 0` 的模板中查找覆盖率 >= 80% 的达标模板
3. **第三步**（降级策略）：如果没有 >= 80% 的模板，降级推荐：
   - 覆盖率最高的模板（1 个）
   - 热度最高的模板（1 个，如果与覆盖率最高不同）
   - 如果是同一个模板，则只输出 1 个

**工具返回结构**：

```json
{
  "has_qualified_templates": true/false,
  "matched_templates": [...],
  "fallback_reason": "降级原因（仅当 has_qualified_templates = false 时）"
}
```

**返回字段说明**：

| 字段 | 说明 |
|------|------|
| `has_qualified_templates` | 是否有覆盖率 >= 80% 的达标模板 |
| `matched_templates` | 达标模板列表，或降级推荐的模板（最多 2 个） |
| `fallback_reason` | 降级原因说明（仅当无达标模板时返回） |

#### 2.2 LLM 可用性与缺口分析

**目标**：对 2.1 返回的模板，LLM 结合用户问题与模板的完整指标构成，分析该模板的实际可用性，并对缺失指标（`missing_indicator_aliases`）给出具体的影响说明与补充建议。

**执行规则**：
- 若 `has_qualified_templates = false`（降级推荐）：**必须**向用户说明"未找到覆盖率 >= 80% 的达标模板，以下为参考推荐"
- 覆盖率 = 100%（`missing_indicator_aliases` 为空）→ 标记为"完整覆盖"，跳过缺口分析
- 覆盖率 < 100%（`missing_indicator_aliases` 不为空）→ 执行缺口分析

**LLM 可用性与缺口分析 Prompt**：

```
你是银行数据分析专家。请基于用户问题，对以下模板的指标构成进行分析，
判断该模板的实际可用性，并说明缺失指标的影响与补充建议。

【用户原始问题】
{user_question}

【用户确认的分析维度及关联指标】
{for dim in analysis_dimensions}
- 「{dim}」关联指标: {dim.matched_indicators}
{/for}

【模板信息】
模板名称: {template.template_alias}
模板描述: {template.template_description}
覆盖率: {template.coverage_ratio * 100}%

【模板包含的全量指标】
{for ind in template.all_template_indicators}
- {ind.alias}：{ind.description}
{/for}

【缺失指标（用户需求中未被该模板覆盖的部分）】
{for ind in missing_indicators}
- {ind.alias}（类型: 筛选指标/分析指标）：{ind.description}
{/for}

【分析任务】
1. 结合模板的全量指标与用户问题，评估该模板能在多大程度上满足用户需求
2. 针对每个缺失指标，判断其对用户分析的重要程度及缺失影响
3. 给出整体可用性结论与补充建议

【输出格式（JSON）】
{
  "template_id": "...",
  "overall_usability": "可直接使用 / 补充后可用 / 缺口较大建议谨慎",
  "usability_summary": "整体可用性的一句话说明，供用户快速判断",
  "missing_indicator_analysis": [
    {
      "indicator_alias": "缺失指标别名",
      "importance": "核心 / 辅助 / 可忽略",
      "impact": "缺失该指标对用户分析的具体影响",
      "supplement_suggestion": "补充建议（无需补充则填'无'）"
    }
  ]
}
```

**可用性等级说明**：

| 可用性等级 | 含义 | 展示标记 |
|-----------|------|---------|
| 可直接使用 | 缺失指标均为辅助或可忽略级别，不影响核心分析 | ✅ |
| 补充后可用 | 缺失部分辅助指标，在模板基础上补充后可满足需求 | 🔧 |
| 缺口较大建议谨慎 | 缺失核心指标，需较多调整才能满足需求 | ⚠️ |

## 核心工具清单

### 阶段 1（主题定位与指标裁决）

| 步骤 | 工具 | 功能 | 路径 |
|------|------|------|------|
| 1.1 主题统计聚合 | `aggregate_themes_from_indicators` | 从指标列表聚合候选主题（按频次排序） | A |
| 1.2 层级导航 | `get_sectors_from_root` | 获取"自主分析"下的所有板块 | B |
| 1.2 层级导航 | `get_sector_themes` | 批量获取指定板块下所有 THEME（含完整路径，一次查询直达所有深度） | B |
| 1.3 全量指标补全 | `get_theme_filter_indicators` | 获取主题下全量筛选指标 | - |
| 1.3 全量指标补全 | `get_theme_analysis_indicators` | 获取主题下全量分析指标 | - |

> 路径 A = 统计聚合（基于指标归属），路径 B = 层级导航（基于 ontology 层级结构）

### 阶段 2（模板推荐）

| 工具 | 功能 |
|------|------|
| `get_theme_templates_with_coverage` | 获取主题模板 + 覆盖率计算（含模板全量指标，LLM 缺口分析用） |

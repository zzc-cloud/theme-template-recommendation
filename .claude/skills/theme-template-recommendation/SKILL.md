# 魔数师主题和模板推荐

## 概述

本 Skill 基于用户自然语言问题，推荐"魔数师"平台的主题（THEME）和模板（TEMPLATE）。

**核心价值**：
- **需求澄清**：通过向量化语义搜索将用户问题映射到魔数师指标，通过人机协作确认映射结果
- **主题推荐**：推荐合适的业务主题
- **指标推荐**：推荐主题下可勾选的核心指标
- **模板推荐**：推荐可直接使用的透视分析/万能查询模板

---

## 输入参数

**用户问题**（必填）：自然语言表述的数据分析需求

```
用户问题: "我想分析南京分行的小微企业贷款风险"
```

**可选参数**：
- `top_k_themes`: 返回主题数量（默认 3）
- `top_k_templates`: 每种类型返回模板数量（默认 5）

---

## 核心概念：统一映射到指标

**所有用户词组都映射到魔数师指标**，没有"维度"的概念。

| 用户词组 | 映射到的指标 | 说明 | 是否需要用户确认 |
|----------|--------------|------|----------------|
| "**分行" | **二级账务机构名称** | 机构筛选指标 | ❌ 自动应用 |
| "**年" | **数据日期** | 时间筛选指标 | ❌ 自动应用 |

**关键理解**：
- **筛选值**（机构、时间、地区）：直接映射为筛选指标，**自动应用，无需用户确认**
- **分析概念**：向量搜索后由 LLM 精炼为关键词，用户确认的是**分析意图（关键词）**而非具体指标
- 所有映射结果都是魔数师指标，只是用途不同（分析度量 vs 筛选条件）

## 执行流程

### 阶段 0：需求澄清

**目标**：将用户问题映射到魔数师指标，通过人机协作确认分析意图。

**处理原则**：
- **筛选值**（机构、时间、地区）：直接映射，**自动应用，无需用户确认**
- **分析概念**：向量搜索 → LLM 精炼关键词 → 用户确认关键词

**⚠️ 关键词精炼的用途**：向量搜索返回相似结果后，LLM 分析这些结果生成更精确的组合关键词，用户确认的是分析意图而非具体指标。如果所有搜索结果相似度 < 0.6，LLM 精炼返回空，要求用户换关键词。

      "matched_indicators": [

          "indicator_id": "指标ID",
          "alias": "指标别名",

      ]
    }
  ],
  "reasoning": "精炼逻辑说明"
}

- 精炼关键词数量控制在 1-5 个
- 每个关键词至少关联 1 个指标
- 优先选择相似度 > 0.75 的指标
- 如果所有结果相似度都很低（如 < 0.5），请标注"低置信度"

**阶段 0 → 阶段 1 传递内容**：

### 阶段 1：指标信息获取

**目标**：基于阶段 0 输出的确认关键词及关联指标，获取完整信息用于后续裁决。

```
┌─────────────────────────────────────────────────────────────┐
│  [1.1] 使用阶段 0 的确认关键词                                   │
├─────────────────────────────────────────────────────────────┤
│  输入：refined_keywords + matched_indicators（来自阶段 0）    │
│                                                              │
│  对每个指标（从 matched_indicators 获取）：                    │
│  1. get_indicator_full_path(indicator_id) → 业务路径         │
│     → 从 path_nodes 中提取 THEME 信息                        │
│  2. 【语义增强】获取指标的业务描述：                          │
│     - get_indicator_field_mapping(indicator_id) → 字段映射   │
│     - get_table_terms(schema, table) → 术语描述              │
│     - 过滤：只保留与指标字段名匹配的术语                      │
└─────────────────────────────────────────────────────────────┘
```

**语义增强规则**：
- **优先使用术语描述**（term_description）：术语描述通常比指标描述更详细
- **降级策略**：如果术语无描述，使用指标描述（indicator_desc）；如果都无描述，只显示别名
- **术语过滤**：字段名与指标字段名完全匹配（大小写不敏感）
  - 例如：指标字段 `cfc_loan_bal` 匹配术语字段 `CFC_LOAN_BAL`

**效率优化**：
- 只对最终推荐给用户的指标进行语义增强（通常 5-15 个）
- 同一张表的术语只获取一次（缓存机制）

### 阶段 2：主题聚合与决策

**目标**：从匹配指标中聚合 THEME，决策出推荐主题。

**执行步骤**：

```

│  [2.1] THEME 统计聚合                                        │
├─────────────────────────────────────────────────────────────┤

│                                                              │
│  统计逻辑：                                                   │
│  - 统计各 THEME 出现频次                                     │
│  - 按频次排序                                                │
│  - 选择 Top 3 THEME 作为推荐主题                             │
│                                                              │
│  示例结果：                                                   │
│  THEME.DS_SSA_RISK。对公贷款借据: 15 个指标                   │
│  THEME.DS_SSA_RISK。对公贷款客户: 8 个指标                    │
│  THEME.DS_SSA_RISK。贷款质量: 5 个指标                       │
└─────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  [2.2] 为每个推荐主题筛选覆盖用户问题的指标                   │
├─────────────────────────────────────────────────────────────┤
│  对每个推荐主题：                                             │
│  - 筛选属于该主题的匹配指标                                   │
│  - 根据用户问题语义，选择能够覆盖用户问题的指标                │
│  - 指标数量不设上限，取决于用户问题的复杂度                    │
│    - 简单问题：可能只需要 2-3 个指标                          │
│    - 复杂问题：可能需要 10+ 个指标                            │
│  - 输出指标 ID、别名、描述                                    │
│                                                              │
│  关键原则：推荐的指标必须能够完整覆盖用户的语义诉求             │
└─────────────────────────────────────────────────────────────┘
```

**为什么采用主题优先？**：
1. 模板数量太多（75,257 个），直接推荐噪音大
2. 相似模板多（不同用户在同一主题下创建相近模板）
3. 主题是天然的过滤器，缩小搜索空间

---

### 阶段 3：模板推荐（两阶段：程序化初筛 + LLM 精细化判定）

**目标**：在推荐主题范围内，推荐可直接使用的模板。对于覆盖率 < 80% 的候选模板，提供用户主动触发的 LLM 精细化分析。

**核心原则**：
- **覆盖率 >= 80%**：程序化直接推荐
- **覆盖率 < 80%**：输出为候选模板，由用户决定是否需要 LLM 深度分析
- LLM 分析是用户**主动触发**的动作，与覆盖率阈值完全解耦，体验一致

**执行步骤**：

```
┌─────────────────────────────────────────────────────────────────┐
│  [3.1] 程序化初筛                                               │
├─────────────────────────────────────────────────────────────────┤
│  对每个推荐主题调用:                                             │
│  get_theme_templates_with_coverage(                              │
│    theme_id="...",                                               │
│    matched_indicator_ids=[从 refined_keywords 提取的指标ID列表],   │
│    template_type=None,                                            │
│    top_k=20                                                       │
│  )                                                               │
│                                                                   │
│  覆盖率计算:                                                      │
│  coverage = 匹配指标数 / 用户需要的指标数（来自 refined_keywords）  │
│                                                                   │
│  分类输出:                                                       │
│  - coverage >= 80% → ✅ 推荐模板（可直接使用）                    │
│  - coverage < 80% → 📋 候选模板（按覆盖率排序）                   │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────────┐
│  [3.2] 始终询问用户                                              │
├─────────────────────────────────────────────────────────────────┤
│  无论是否有 >= 80% 的模板，都输出:                               │
│  - 推荐模板（覆盖率 >= 80%）                                     │
│  - 候选模板列表（覆盖率 < 80%，按覆盖率排序）                     │
│  - 询问：是否需要 LLM 深度分析候选模板？                         │
│                                                                   │
│  💡 用户主动触发后，进入 LLM 精细化判定                          │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼ (用户主动触发)
┌─────────────────────────────────────────────────────────────────┐
│  [3.3] LLM 精细化判定                                           │
├─────────────────────────────────────────────────────────────────┤
│  对每个候选模板:                                                  │
│  1. 获取模板完整信息（调用 get_template_indicators）             │
│  2. 构建 LLM 分析 prompt                                        │
│  3. LLM 分析并输出结构化判定结果                                 │
│                                                                   │
│  LLM 分析内容:                                                   │
│  - 业务场景匹配度: 模板 vs 用户需求                               │
│  - 数据粒度匹配度: 明细级 vs 汇总级                              │
│  - 最终判定: 推荐/不推荐 + 理由                                   │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

**设计理由**：
- **体验一致**：无论覆盖率如何，用户每次都看到相同的结果格式和询问
- **逻辑简单**：不需要根据覆盖率决定输出什么
- **真正的用户自主**：用户自己决定是否需要深入分析，不管覆盖率是多少

## LLM 精炼关键词 Prompt

当向量搜索返回分析概念的指标候选后，使用以下 prompt 进行精炼：

```
你是银行数据分析专家。请根据用户的原始问题，从向量搜索结果中精炼出最准确的分析关键词。

【用户原始问题】
{user_question}

【向量搜索结果】
{for keyword, indicators in analysis_search_results}
关键词：「{keyword}」的搜索结果（Top 10，按相似度排序）：
{for indicator in indicators}
  - {indicator.alias}（相似度: {indicator.score}）
    描述: {indicator.description}
{/for}
{/for}

【分析要求】
1. 理解用户原始问题的分析意图
2. 从搜索结果中选择最相关的指标组合
3. 将选中的指标组合精炼为 1-3 个精确的分析关键词/词组
4. 每个精炼关键词应能清晰表达一个分析维度

【精炼规则】
- 合并相似概念，避免重复（例如："小微企业贷款"+"风险" → "小微企业贷款风险"）
- 去歧义：如果多个词指向同一分析意图，合并为一个关键词
- 补充遗漏：如果用户意图未被覆盖，适当补充
- 每个关键词需要附带其关联的指标列表
- 如果所有搜索结果相似度 < 0.6，返回空列表，说明无法匹配

【输出格式】
请以 JSON 格式输出精炼后的关键词列表：

{
  "refined_keywords": [
    {
      "keyword": "精炼后的关键词",
      "description": "该关键词的分析意图说明",
      "source_keywords": ["来源的原始关键词"],
      "matched_indicators": [
        {
          "indicator_id": "指标ID",
          "alias": "指标别名",
          "score": 0.88
        }
      ]
    }
  ],
  "reasoning": "精炼逻辑说明"
}

【注意事项】
- 精炼关键词数量控制在 1-5 个
- 每个关键词至少关联 1 个指标
- 优先选择相似度 > 0.75 的指标
- 如果所有结果相似度都很低（如 < 0.5），请标注"低置信度"
```

## LLM 精细化分析 Prompt

当用户触发 LLM 分析时，使用以下 prompt 模板进行分析：

```
你是银行数据分析专家。请分析以下候选模板与用户需求的业务匹配度。

【用户需求】
{user_question}

【用户确认的关键词及关联指标】
{for keyword in refined_keywords}
- {keyword.keyword}
  关联指标: {keyword.matched_indicators}
{/for}

【候选模板】
{for template in candidate_templates}
- 模板名称: {template.alias}
- 模板描述: {template.description}
- 覆盖率: {template.coverage_ratio}%
- 包含指标: {template.indicators}
{/for}

【分析要求】
对每个候选模板，分析：
1. 业务场景匹配度：该模板的业务场景是什么？与用户需求是否匹配？
2. 数据粒度匹配度：该模板是明细级还是汇总级？
3. 指标覆盖分析：该模板包含的指标是否能满足用户核心需求？
4. 最终判定：推荐/不推荐，并说明理由

【输出格式】
{
  "template_id": "...",
  "business_match_score": 1-5,
  "granularity_match": "明细级/汇总级/不适用",
  "reasoning": "判定理由",
  "recommendation": "推荐/不推荐",
  "supplementary_indicators": ["补充指标列表"]
}
```

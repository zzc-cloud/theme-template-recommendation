# 魔数师主题和模板推荐

## 概述

本 Skill 基于用户自然语言问题，推荐"魔数师"平台的主题（THEME）和模板（TEMPLATE）。

**核心价值**：
- **需求澄清**：通过向量化语义搜索将用户问题映射到魔数师指标，通过人机协作确认映射结果
- **主题推荐**：推荐合适的业务主题
- **指标推荐**：推荐主题下可勾选的核心指标
- **模板推荐**：推荐可直接使用的透视分析/万能查询模板

---

## 输入参数

**用户问题**（必填）：自然语言表述的数据分析需求

```
用户问题: "我想分析南京分行的小微企业贷款风险"
```

**可选参数**：
- `top_k_themes`: 返回主题数量（默认 3）
- `top_k_templates`: 每种类型返回模板数量（默认 5）

---

## 核心概念：统一映射到指标

**所有用户词组都映射到魔数师指标**，没有"维度"的概念。

| 用户词组 | 映射到的指标 | 说明 | 是否需要用户确认 |
|----------|--------------|------|----------------|
| "**分行" | **二级账务机构名称** | 机构筛选指标 | ❌ 自动应用 |
| "**年" | **数据日期** | 时间筛选指标 | ❌ 自动应用 |

**关键理解**：
- **筛选值**（机构、时间、地区）：直接映射为筛选指标，**自动应用，无需用户确认**
- **分析概念**：向量搜索后经迭代精炼收敛，收敛后的搜索词即为分析维度，用户确认的是**分析意图**而非具体指标
- 所有映射结果都是魔数师指标，只是用途不同（分析度量 vs 筛选条件）

## 执行流程

### 阶段 0：需求澄清

**目标**：将用户问题映射到魔数师指标，通过人机协作确认分析意图。

**处理原则**：
- **筛选值**（机构、时间、地区）：直接映射，**自动应用，无需用户确认**
- **分析概念**：向量搜索 → 迭代精炼循环收敛 → 收敛后搜索词直接作为分析维度 → 用户确认

#### 0.1 原始词组提取

从用户问题中提取业务相关的词组，不做任何分类或加工：

```
提取示例：
- "南京分行"
- "小微企业贷款"
- "风险"
```

#### 0.2 词组分类与分类处理

**内部执行，不展示给用户。**

1. **LLM 语义分类**，对每个词组判断类型：
   - **筛选值类型**：
     - 机构名称（分行/支行）："南京分行"、"某某支行"
     - 时间词（年/月/日）："2024年"、"今年"、"上个月"
     - 地区/行政区划："江苏省"、"浙江省"、"南京"
   - **分析概念类型**：
     - 业务术语："小微企业贷款"、"贷款余额"、"不良率"
     - 分析意图词："风险"、"增长"、"占比"

2. **分类处理**（静默执行）：
   - 筛选值类型 → 直接映射到指标，存入 `filter_indicators`
   - 分析概念类型 → 调用 `search_indicators_by_vector`，进入 0.3 迭代精炼循环

**内部数据结构示例**：

```
filter_indicators = [
  { indicator_id: "二级账务机构名称", value: "南京分行" }
]

search_results_v1 = {
  "小微企业贷款": [
    { indicator_id: "...", alias: "借据余额", score: 0.88 },
    { indicator_id: "...", alias: "贷款本金", score: 0.75 }
  ],
  "风险": [
    { indicator_id: "...", alias: "操作风险", score: 0.82 }
  ]
}
```

#### 0.3 迭代精炼循环（自动执行，用户不感知）

**目标**：通过多轮搜索 + LLM 评估，将分析概念收敛为高质量的候选指标集合。**收敛后的搜索词直接作为分析维度**，无需额外归纳步骤。

**⚠️ 核心原则：意图锚定（最重要）**

在整个迭代精炼过程中，必须始终以用户原始��题的**核心意图**作为锚点。
**修正搜索词时不能偏离原始意图**，否则会导致推荐结果与用户需求不符。

**锚定验证规则**：
- 修正后的搜索词必须**保留原始问题中的核心语义单元**（见下方定义）
- 修正后的搜索词**不能引入原始问题中没有的全新语义域**
- 若修正词与原始意图偏离度超过 50%，触发**偏离警告**，放弃该修正

**⚠️ 核心语义单元���义（关键）**

**语义单元 ≠ 字面关键词 token**。偏离度必须基于**语义单元**计算，而非逐字拆分。

语义单元是用户原始问题中承载独立业务含义的最小不可拆分片段。判定规则：
1. **专有业务术语**是独立语义单元，不可拆分。例如："结算客户"、"个人存款"、"小微企业贷款"
2. **独立概念词**各为独立语义单元。例如："转账"、"客户"、"风险"、"余额"
3. 语义单元的识别以**业务语义完整性**为准，而非字面包含关系

**错误示例**（字面 token 级判定）：
- 原始概念："结算客户数" → 拆分为 ["结算", "客户", "数"]
- 修正词："结算账户客户数" → 包含"结算"✓ "客户"✓ "数"✓ → 判定偏离度 = 0% → **错误地放行**
- 实际问题："结算客户"是一个不可拆分的业务术语（指办理结算业务的客户），修正为"结算账户客户数"后语义从"业务实体"偏移到了"账户统计"

**正确示例**（语义单元级判定）：
- 原始概念："结算客户数" → 语义单元 = ["结算客户", "数"]
- 修正词："结算账户客户数" → 包含"结算客户"？❌（被拆成了"结算"+"账户客户"）
- 偏离度 = 1/2 = 50% → 触发偏离警告 → **正确拦截**

**循环逻辑**：

```
输入：
  - user_question: 用户原始问题
  - original_concepts: 原始分析概念（锚点，不随迭代变化）
  - original_semantic_units: 原始概念的语义单元列表（锚点，不随迭代变化）
  - analysis_concepts: 当前待搜索的分析概念词组列表
  - search_results: 各概念的搜索结果
  - converged_concepts: 已收敛概念集合（初始为空）
  - unconverged_concepts: 未收敛概念集合（初始为 analysis_concepts）
  - iteration_log: 迭代过程记录

WHILE 轮次 < 5:
  轮次 +1
  1. 对 unconverged_concepts 中的每个概念调用 search_indicators_by_vector
  2. 初始收敛判定：Top-1 >= 0.80 的概念标记为"候选收敛"
  3. 【收敛反向语义验证】对每个"候选收敛"的概念执行：
     - 检查其 Top-1 指标的别名/描述，是否真正属于 original_semantic_units 的语义范围
     - 验证问题："这个 Top-1 指标，是否是在回答用户原始问题中的业务概念？"
     - 若 Top-1 指标与原始语义单元语义一致 → 确认收敛，移入 converged_concepts
     - 若 Top-1 指标与原始语义单元语义偏离 → 标记为"虚假收敛"，将该 Top-1 指标加入该概念的"排除列表"，留在 unconverged_concepts
  4. 【剔除后重新评估收敛】⚠️ 关键步骤
     - 对每个仍有虚假收敛记录的概念，用其"排除列表"过滤搜索结果
     - 对过滤后的剩余结果，重新执行两步收敛判定：
       - 若 Top-1(排除后) >= 0.80 → 确认收敛，移入 converged_concepts（记录"排除后收敛"）
       - 若 Top-1(排除后) < 0.80 → 保持 unconverged，进入下一步
     - 重复此步骤直到没有新收敛的概念（支持多级排除）
  5. 若 unconverged_concepts 为空 → 退出循环（正常收敛出口）
  6. 若轮次已达 5 → 退出循环（超时出口）
  7. 【意图锚定检查】评估当前收敛进度：
     - 若已收敛概念 >= 50%，且核心概念已收敛 → 允许未收敛概念进入低置信度
     - 若收敛率 < 50% → 继续迭代修正
  8. 对 unconverged_concepts 调用 LLM 生成下一轮修正搜索词
  9. 【意图锚定验证】检查修正词是否偏离原始意图（基于语义单元）：
     - 修正词是否保留了 original_semantic_units 中的核心语义单元？
     - 修正词是否引入原始问题中没有的全新语义域？
     - 计算偏离度时使用语义单元（而非字面 token）
     - 若偏离度过高 → 放弃修正，强制收敛或标记低置信度
  10. 若修正词与当前搜索词完全相同 → 退出循环（自然收敛）
  11. 用修正词替换 unconverged_concepts，进入下一轮
END WHILE

输出：
  - converged_search_terms: 收敛后的搜索词列表（即分析维度）
  - final_search_results: 每个搜索词对应的候选指标集合
  - low_confidence_concepts: 未收敛但已尽力收敛的概念
  - deviation_warnings: 意图偏离警告（如有）
  - false_convergence_warnings: 虚假收敛警告（如有）
```

**收敛判断标准（两步验证）**：

**第一步：初始收敛条件（满足任一即标记为"候选收敛"）**：

| 条件 | 说明 | 判断时机 |
|------|------|----------|
| 该概念 Top-1 相似度 >= 0.80 | 语义命中精准 | 每轮搜索后，逐概念判断 |
| 该概念 Top-5 中 >= 3 个指标属于同一 THEME | 主题命中集中 | 每轮搜索后，主题聚合判断 |
| 本轮修正词与当前搜索词完全相同 | 自然收敛 | LLM 生成修正词后判断 |

**第二步：反向语义验证（防止虚假收敛）**：

对每个"候选收敛"的概念，必须验证 Top-1 指标是否真正属于用户原始意图的语义范围。

| 验证项 | 说明 | 示例 |
|--------|------|------|
| Top-1 指标是否回答了原始语义单元的业务问题 | 不能仅因为相似度高就认为收敛 | ✅ 原始意图"结算客户" → Top-1"结算客户标志"(0.86) → 同属结算客户业务 → 确认收敛 |
| | | ❌ 原始意图"结算客户" → Top-1"存款账户数"(0.86) → 从"业务实体"偏移到"账户统计" → 虚假收敛 |

**虚假收敛处理**：
- 将概念退回 unconverged_concepts，并将该 Top-1 指标加入该概念的"排除列表"
- 在 iteration_log 中记录虚假收敛原因和排除的指标
- 立刻进入步骤 4（剔除后重新评估收敛），用排除列表过滤后再判定
- 若连续 2 次触发虚假收敛 → 强制使用原始概念词作为搜索词，不再修正

**超时出口处理（低置信度）**：

5 轮结束后 unconverged_concepts 仍有未收敛概念（即存在 Top-1 相似度 < 0.80 的概念），进入低置信度流程：

1. **收敛率评估**：计算 `已收敛概念数 / 原始概念总数`
   - 收敛率 >= 50%，且核心概念已收敛 → 允许未收敛概念进入低置信度，继续执行
   - 收敛率 < 50% → 向用户说明"匹配质量不足"，建议重新描述问题

2. **低置信度展示**：展示已收敛维度（默认勾选）和未收敛维度（带警告、默认不勾选），由用户决定是否纳入未收敛维度继续执行，或重新描述换词

> **核心概念判定**：核心概念是指从用户原始问题中提取的、承载用户主要分析意图的概念。
> 例如用户问"分析大额转账客户明细"，核心意图是"转账交易"+"客户"，而非"明细"。若"转账"相关指标已收敛，即使"明细"未收敛，也可视为核心概念已收敛。

**每轮 LLM 修正搜索词生成 Prompt**（仅对未收敛概念执行）：

```
你是银行数据分析专家，正在帮助用户精确定位分析指标。

【用户原始问题】
{user_question}

【当前轮次】第 {round} 轮

【原始语义单元】
{for concept, units in original_semantic_units}
「{concept}」的语义单元: {units}
{/for}
说明：语义单元是不可拆分的业务含义片段。偏离度基于语义单元计算，不是逐字判断。

【已收敛概念及结果】
{for concept, results in converged_concepts}
✓ 「{concept}」已收敛（Top-1 相似度 >= 0.80）
  Top-1 结果: {results[0].alias}（相似度: {results[0].score}）
{/for}

【未收敛分析概念及搜索结果】
{for concept, results in unconverged_concepts}
分析概念：「{concept}」
排除指标（如有）: {excluded_indicators}  ← 这些指标因虚假收敛已被排除，不能作为收敛依据
搜索结果 Top-5：
  {for r in results[:5]}
  - {r.alias}（相似度: {r.score}）描述: {r.description}
  {/for}
{/for}

【⚠️ 意图锚定约束（最重要）】

修正搜索词时，必须严格遵守以下约束：

1. **必须保留核心语义单元**：修正词必须保留原始语义单元的完整业务含义
   - 例如：用户问"查询各分行结算客户数"，语义单元 = ["结算客户", "数"]
   - "结算客户"是一个不可拆分的业务术语（指办理结算业务的客户），必须作为整体保留
   - 绝对不能拆开语义单元去匹配。❌ "结算账户客户数" → 拆开了"结算客户"，引入了"账户"
   - ✅ "结算客户标志"、"结算客户统计" → 保留了"结算客户"这一完整语义单元

2. **禁止引入新语义域**：修正词不能引入原始问题中没有的全新语义域
   - ❌ 原始意图"结算客户" → 修正成"存款账户数"（从"业务实体"偏移到"账户统计"）
   - ❌ 原始意图"结算客户" → 修正成"结算账户客户数"（引入"账户"语义域）
   - ✅ 原始意图"结算客户" → 可修正为"结算客户标志"、"结算客户信息"

3. **偏离度判定（基于语义单元）**：若修正词与原始意图偏离度超过 50%，必须放弃修正
   - 判定方法：检查修正词是否保留了原始语义单元的完整业务含义
   - 对每个语义单元判断：修正词是否保持了该单元的语义完整性？
   - 偏离度 = 被破坏的语义单元数 / 原始语义单元总数
   - 若偏离度 > 50%，标记为"DEVIATION_ABORT"并跳过该概念

【诊断任务】
针对每个未收敛的分析概念（Top-1 相似度 < 0.80），诊断问题并生成修正搜索词：
- 搜索词过于宽泛？（结果太杂）→ 收缩
- 搜索词过于口语化？（结果偏离）→ 替换
- 搜索词缺少限定词？（召回不足）→ 扩展
- 多个搜索词指向同一意图？→ 合并

【输出格式（JSON）】
{
  "corrections": [
    {
      "original": "原搜索词",
      "action": "扩展/收缩/替换/合并",
      "corrected": "修正后搜索词（若偏离度过高则为 DEVIATION_ABORT）",
      "semantic_unit_check": {
        "original_units": ["语义单元1", "语义单元2"],
        "preserved": ["保留的单元"],
        "broken": ["被破坏的单元（如有）"],
        "deviation_score": 0.0-1.0,
        "new_domains_introduced": ["引入的新语义域（如有）"]
      },
      "reason": "修正原因"
    }
  ],
  "normalized_question": "规范化后的问题描述（若本轮后预计全部收敛则输出）"
}
```

#### 0.4 用户确认分析维度 + 勾选引导

基于 0.3 所有的分析概念（收敛的和未收敛的分析概念）及其关联候选指标，展示给用户确认。

**关键说明**：
- **分析维度 = 0.3 收敛后的搜索词**，无需额外归纳，直接展示
- 指标仅作为辅助说明，帮助用户理解该维度的含义
- 筛选值不展示在选项中，仅在说明中标注"自动应用"

**⚠️ 勾选引导（核心改进）**：

多个分析维度同时勾选时，可能出现"主题交叉干扰"问题，导致推荐效果下降。

**核心改进**：通过查询 Neo4j 获取每个维度的真实 theme_id，基于**相似度加权的 Jaccard 相似度**判断维度间主题交叉程度，避免仅凭指标别名猜测主题导致的误判。

**处理流程（3 步）**：

1. **查 Neo4j（批量，一次请求）**：
   - 对每个分析维度，取其 Top-20 候选指标（来自 0.3 搜索结果，含相似度分数）
   - 合并去重后，一次调用 `batch_get_indicator_themes(indicator_ids)` 获取所有指标的主题归属
   - 按维度分组聚合：每个维度命中的所有 theme 及其**对该维度的贡献权重**（取指标最大相似度）

2. **算加权 Jaccard**：计算维度两两之间的加权 Jaccard 相似度
   - 传统 Jaccard：`J = |交集| / |并集|`（所有 theme 等权重）
   - **加权 Jaccard**：`J = Σ(min(w_A, w_B)) / Σ(max(w_A, w_B))`
     - `w_A`：theme 对维度 A 的贡献权重（= 映射到该 theme 的指标相似度之和）
     - `w_B`：theme 对维度 B 的贡献权重
   - Jaccard >= 0.5 → 主题高度重叠，两个维度互补性强（可同时勾选）
   - Jaccard < 0.5  → 主题几乎无交集，存在主题交叉干扰（建议优先勾选核心维度）

**加权 Jaccard 计算示例**：

| Theme | 维度A权重(w_A) | 维度B权重(w_B) | min | max |
|-------|----------------|----------------|-----|-----|
| 个人存款 | 0.92 | 0.85 | 0.85 | 0.92 |
| 小微贷款 | 0.88 | 0.38 | 0.38 | 0.88 |
| 企业贷款 | 0.31 | 0.92 | 0.31 | 0.92 |
| **合计** | - | - | **1.54** | **2.72** |

```
加权 Jaccard = 1.54 / 2.72 = 0.57  (>= 0.5，两个维度互补)
```

**⚠️ 权重衰减规则**：
- 相似度 >= 0.80 → 全权重（1.0）
- 相似度 >= 0.50 → 半权重（0.5）
- 相似度 < 0.50 → 极低权重（0.1）或排除

3. **LLM 引导**：将 theme 信息 + 加权 Jaccard 矩阵注入 Prompt，LLM 综合分析后生成引导建议

**引导输出结构**：

```
dimension_guidance: {
  "has_conflict": true/false,          // 是否存在主题交叉（基于加权 Jaccard < 0.5 判定）
  "recommended_first": ["维度A"],       // 建议优先勾选的核心维度
  "conflict_analysis": "维度A命中了X个主题（加权分X.X），维度B命中了Y个主题（加权分Y.Y），加权Jaccard=xx（<0.5=有冲突），...",
  "selection_advice": "勾选建议（1-2句话）",
  "dimension_analysis": [
    {
      "dimension": "维度名",
      "matched_themes": [
        {"theme": "主题1", "weight": 0.92},
        {"theme": "主题2", "weight": 0.88}
      ],                                   // 命中的主题及贡献权重
      "theme_count": 4,                    // 命中主题数量
      "total_weight": 1.80,               // 权重总和（用于排序）
      "independence_score": 0.0-1.0,       // 独立性得分（LLM 给出）
      "core_concept_score": 0.0-1.0,       // 核心概念得分（LLM 给出）
      "recommendation": "优先/可选/建议后选"
    }
  ]
}
```

**异常处理**：若 LLM 调用失败（网络/解析错误），`dimension_guidance` 为 `None`，跳过引导步骤继续执行后续流程。

> 告知用户"建议优先勾选最核心的维度，全选不一定最优"；若 `has_conflict = false`，
> 说明各维度主题一致或互补，无需额外引导。

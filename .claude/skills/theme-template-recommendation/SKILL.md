# 魔数师主题和模板推荐

## 概述

本 Skill 基于用户自然语言问题，推荐"魔数师"平台的主题（THEME）和模板（TEMPLATE）。

**核心价值**：
- **需求澄清**：通过向量化语义搜索将用户问题映射到魔数师指标，通过人机协作确认映射结果
- **主题推荐**：推荐合适的业务主题
- **指标推荐**：推荐主题下可勾选的核心指标
- **模板推荐**：推荐可直接使用的透视分析/万能查询模板

---

## 输入参数

**用户问题**（必填）：自然语言表述的数据分析需求

```
用户问题: "我想分析南京分行的小微企业贷款风险"
```

**可选参数**：
- `top_k_themes`: 返回主题数量（默认 3）
- `top_k_templates`: 每种类型返回模板数量（默认 5）

---

## 核心概念：统一映射到指标

**所有用户词组都映射到魔数师指标**，没有"维度"的概念。

| 用户词组 | 映射到的指标 | 说明 | 是否需要用户确认 |
|----------|--------------|------|----------------|
| "**分行" | **二级账务机构名称** | 机构筛选指标 | ❌ 自动应用 |
| "**年" | **数据日期** | 时间筛选指标 | ❌ 自动应用 |

**关键理解**：
- **筛选值**（机构、时间、地区）：直接映射为筛选指标，**自动应用，无需用户确认**
- **分析概念**：向量搜索后经迭代精炼收敛，用户确认的是**分析意图**而非具体指标
- 所有映射结果都是魔数师指标，只是用途不同（分析度量 vs 筛选条件）

## 执行流程

### 阶段 0：需求澄清

**目标**：将用户问题映射到魔数师指标，通过人机协作确认分析意图。

**处理原则**：
- **筛选值**（机构、时间、地区）：直接映射，**自动应用，无需用户确认**
- **分析概念**：向量搜索 → 迭代精炼循环收敛 → 用户确认

#### 0.1 原始词组提取

从用户问题中提取业务相关的词组，不做任何分类或加工：

```
提取示例：
- "南京分行"
- "小微企业贷款"
- "风险"
```

#### 0.2 词组分类与分类处理

**内部执行，不展示给用户。**

1. **LLM 语义分类**，对每个词组判断类型：
   - **筛选值类型**：
     - 机构名称（分行/支行）："南京分行"、"某某支行"
     - 时间词（年/月/日）："2024年"、"今年"、"上个月"
     - 地区/行政区划："江苏省"、"浙江省"、"南京"
   - **分析概念类型**：
     - 业务术语："小微企业贷款"、"贷款余额"、"不良率"
     - 分析意图词："风险"、"增长"、"占比"

2. **分类处理**（静默执行）：
   - 筛选值类型 → 直接映射到指标，存入 `filter_indicators`
   - 分析概念类型 → 调用 `search_indicators_by_vector`，进入 0.3 迭代精炼循环

**内部数据结构示例**：

```
filter_indicators = [
  { indicator_id: "二级账务机构名称", value: "南京分行" }
]

search_results_v1 = {
  "小微企业贷款": [
    { indicator_id: "...", alias: "借据余额", score: 0.88 },
    { indicator_id: "...", alias: "贷款本金", score: 0.75 }
  ],
  "风险": [
    { indicator_id: "...", alias: "操作风险", score: 0.82 }
  ]
}
```

#### 0.3 迭代精炼循环（自动执行，用户不感知）

**目标**：通过多轮搜索+LLM评估，将分析概念收敛为高质量的候选指标集合。

**循环逻辑**：

```
输入：
  - analysis_concepts: 0.2 输出的分析概念词组列表
  - search_results_v1: 0.2 中第 1 轮向量搜索结果

WHILE 未收敛 AND 轮次 <= 3:
  1. LLM 评估当前搜索结果质量
  2. 若质量达标 → 收敛出口，退出循环
  3. 若质量不达标 → 生成修正搜索词
  4. 调用 search_indicators_by_vector（修正后搜索词）
  5. 合并新旧搜索结果（去重，保留高分候选）
  6. 轮次 +1
END WHILE

输出：
  - final_search_results: 收敛后的候选指标集合
  - iteration_log: 迭代过程记录（用于探查输出展示）
```

**收敛判断标准（满足任一即收敛）**：

| 条件 | 说明 |
|------|------|
| 每个分析概念 Top-1 相似度 > 0.80 | 语义命中精准 |
| 本轮修正搜索词与上轮完全相同 | 自然收敛，无新变化 |
| 本轮新增候选指标数 = 0 | 搜索结果已饱和 |

**超时出口处理（低置信度）**：

3 轮结束后仍有分析概念 Top-1 相似度 < 0.60，进入低置信度流程：
- 停止迭代，不进入用户确认
- 向用户说明哪个分析概念无法匹配
- 提供 2~3 个换词建议
- 等待用户重新描述后，重新进入 0.1

**每轮 LLM 评估 Prompt**：

```
你是银行数据分析专家，正在帮助用户精确定位分析指标。

【用户原始问题】
{user_question}

【当前轮次】第 {round} 轮

【当前分析概念及搜索结果】
{for concept, results in current_search_results}
分析概念：「{concept}」
搜索结果 Top-5：
  {for r in results[:5]}
  - {r.alias}（相似度: {r.score}）描述: {r.description}
  {/for}
{/for}

【评估任务】
1. 覆盖评估：用户意图的哪些部分已被覆盖？哪些缺失？
2. 质量判断：当前结果是否达标？（每个分析概念 Top-1 > 0.80）
3. 若不达标，诊断问题：
   - 搜索词过于宽泛？（结果太杂）→ 收缩
   - 搜索词过于口语化？（结果偏离）→ 替换
   - 搜索词缺少限定词？（召回不足）→ 扩展
   - 多个搜索词指向同一意图？→ 合并
4. 输出修正搜索词

【输出格式（JSON）】
{
  "quality_pass": true/false,
  "coverage_assessment": "已覆盖XX，缺失XX",
  "corrections": [
    {
      "original": "原搜索词",
      "action": "扩展/收缩/替换/合并",
      "corrected": ["修正后搜索词1", "修正后搜索词2"],
      "reason": "修正原因"
    }
  ]
}
```

#### 0.4 用户确认分析维度

基于迭代精炼循环收敛后的候选指标集合，整理**分析维度**供用户确认。

**关键区别**：
- 用户确认的是**分析维度**（分析意图），而非具体指标
- 指标仅作为辅助说明帮助用户理解该维度的含义
- 筛选值不展示在选项中，仅在说明中标注"自动应用"

**AskUserQuestion 配置**：

```
AskUserQuestion:
  questions:
    - question: "以下筛选条件已自动识别，将直接应用："
      header: "筛选条件（自动应用）"
      options:
        - label: "🏦 二级账务机构名称 = '南京分行'"
          value: "FILTER:auto:二级账务机构名称:南京分行"
          description: "自动映射，无需确认"

    - question: "基于您的分析需求，精炼出以下分析维度，请确认："
      header: "分析维度（请确认）"
      multiSelect: true
      options:
        - label: "📊 小微企业贷款风险"
          value: "DIMENSION:小微企业贷款风险"
          description: "关联指标：借据余额、贷款本金、五级分类、不良率、逾期率"

    - question: "请确认或修改分析需求描述："
      header: "需求确认"
      options:
        - label: "\"分析南京分行的小微企业贷款信用风险\"（推荐）"
          value: "normalized:分析南京分行的小微企业贷款信用风险"
```

#### 0.5 基于确认分析维度 → 最终指标

用户确认后：
1. 获取用户确认的分析维度
2. 复用 0.3 迭代循环中收敛保存的 `final_search_results`
3. 汇总输出最终指标列表

**输出内容（传递给阶段 1）**：

```
analysis_dimensions:  用户确认的分析维度及关联指标
filter_indicators:    自动映射的筛选指标
matched_indicators:   分析维度对应的最终指标列表
normalized_question:  用户确认后的问题改写
```

### 阶段 1：主题决策与指标精筛

**目标**：从阶段 0 获取确认的分析维度及关联指标，聚合推荐主题并精筛覆盖用户需求的指标。

#### 1.1 THEME 统计聚合

从所有匹配指标中聚合 THEME 信息：
1. 调用 `aggregate_themes_from_indicators(matched_indicators)` 批量获取指标的主题信息
2. 统计各 THEME 出现频次并排序
3. 选择 Top 3 THEME 作为初始推荐主题

**输出示例**：
```json
{
  "candidate_themes": [
    {
      "theme_id": "THEME.xxx",
      "theme_alias": "个人存款",
      "frequency": 5,
      "matched_indicator_ids": ["INDICATOR.xxx", ...]
    }
  ]
}
```

#### 1.2 指标补全

对每个推荐主题执行指标补全操作：

**步骤 1.2.1**：调用 `get_theme_filter_indicators(theme_id)` 获取主题下的全量筛选指标

**筛选指标识别规则**（在 MCP 工具内部实现）：
- **时间筛选指标**：指标别名包含 `数据日期` 或 `ETL数据日期`
- **机构筛选指标**：指标别名包含 `机构名称`、`机构编号`、`管理机构名称`、`管理机构编号`、`账务机构名称`、`账务机构编号`

**步骤 1.2.2**：调用 `get_theme_analysis_indicators(theme_id)` 获取主题下的全量分析指标

分析指标为主题下除去筛选指标外的所有指标。

**步骤 1.2.3**：对分析指标进行语义增强

对每个分析指标执行语义增强：
1. 调用 `get_indicator_field_mapping(indicator_id)` 获取字段映射
2. 调用 `get_table_terms(schema, table)` 获取术语描述
3. 按优先级选择描述：术语描述 > 指标描述 > 别名

**效率优化**：
- 同一张表的术语只获取一次（缓存机制）
- 批量处理指标以减少工具调用次数

#### 1.3 LLM 裁决

**目标**：由 LLM 判断主题是否能支撑用户需求，并精准定位覆盖用户需求的指标。

**子步骤 1.3.1：主题支撑度判断**

对每个候选主题执行支撑度判断，不可支撑则跳过该主题。

```
你是银行数据分析专家，正在评估某个主题是否能支撑用户的分析需求。

【用户原始问题】
{user_question}

【用户确认的分析维度】
{analysis_dimensions}

【候选主题】
主题名称：{theme_alias}
主题路径：{theme_path}

【筛选指标】（用于过滤数据）
{filter_indicators}

【分析指标样例】（共 {analysis_count} 个，以下为前 20 个）
{analysis_indicators_sample}

【评估任务】
1. 该主题的筛选指标是否能满足用户的时间/机构筛选需求？
2. 该主题的分析指标是否覆盖用户的分析维度？
3. 综合判断：该主题能否支撑用户的分析需求？

【输出格式（JSON）】
{
  "can_support": true/false,
  "reason": "判断理由",
  "missing_aspects": ["无法覆盖的方面（如果 can_support=false）"]
}
```

**子步骤 1.3.2：指标精准定位**

对于支撑度判断通过的主题，从全量指标中精准定位覆盖用户需求的指标。

```
你是银行数据分析专家，需要从主题的全量指标中精准定位能覆盖用户需求的指标。

【用户原始问题】
{user_question}

【用户确认的分析维度】
{analysis_dimensions}

【该主题的全量指标】
筛选指标（用于过滤数据）：
{filter_indicators}

分析指标（用于分析度量）：
{analysis_indicators}

【任务】
1. 从筛选指标中，选出用户需要用于过滤的条件
2. 从分析指标中，选出能覆盖用户分析维度的指标

【输出格式（JSON）】
{
  "selected_filter_indicators": [
    {
      "indicator_id": "...",
      "indicator_alias": "...",
      "reason": "选择原因"
    }
  ],
  "selected_analysis_indicators": [
    {
      "indicator_id": "...",
      "indicator_alias": "...",
      "reason": "选择原因"
    }
  ],
  "coverage_assessment": "覆盖了用户需求的哪些方面"
}
```

### 阶段 2：模板推荐

**目标**：在推荐主题范围内，推荐可直接使用的模板。

**核心原则**：
- **覆盖率 >= 80%**：程序化直接推荐
- **覆盖率 < 80%**：输出为候选模板，由用户决定是否需要 LLM 深度分析
- LLM 分析是用户**主动触发**的动作，与覆盖率阈值完全解耦

#### 2.1 程序化初筛

对每个推荐主题调用：

```
get_theme_templates_with_coverage(
  theme_id="...",
  matched_indicator_ids=[从阶段 1 精筛后的指标ID列表],
  template_type=None,
  top_k=20
)
```

覆盖率计算：
```
coverage = 匹配指标数 / 用户需要的指标数（来自阶段 1 精筛结果）
```

分类输出：
- `coverage >= 80%` → ✅ 推荐模板（可直接使用）
- `coverage < 80%` → 📋 候选模板（按覆盖率排序）

#### 2.2 始终询问用户

无论是否有 >= 80% 的模板，都输出：
- 推荐模板（覆盖率 >= 80%）
- 候选模板列表（覆盖率 < 80%，按覆盖率排序）
- 询问：是否需要 LLM 深度分析候选模板？

#### 2.3 LLM 精细化判定（用户主动触发）

对每个候选模板：
1. 调用 `get_template_indicators` 获取模板完整信息
2. LLM 分析业务场景匹配度、数据粒度匹配度
3. 输出结构化判定结果：推荐/不推荐 + 理由

**LLM 分析 Prompt**：

```
你是银行数据分析专家。请分析以下候选模板与用户需求的业务匹配度。

【用户需求】
{user_question}

【用户确认的分析维度及关联指标】
{for dim in analysis_dimensions}
- {dim.dimension}
  关联指标: {dim.matched_indicators}
{/for}

【候选模板】
{for template in candidate_templates}
- 模板名称: {template.alias}
- 模板描述: {template.description}
- 覆盖率: {template.coverage_ratio}%
- 包含指标: {template.indicators}
{/for}

【分析要求】
对每个候选模板，分析：
1. 业务场景匹配度：该模板的业务场景是什么？与用户需求是否匹配？
2. 数据粒度匹配度：该模板是明细级还是汇总级？
3. 指标覆盖分析：该模板包含的指标是否能满足用户核心需求？
4. 最终判定：推荐/不推荐，并说明理由

【输出格式（JSON）】
{
  "template_id": "...",
  "business_match_score": 1-5,
  "granularity_match": "明细级/汇总级/不适用",
  "reasoning": "判定理由",
  "recommendation": "推荐/不推荐",
  "supplementary_indicators": ["补充指标列表"]
}
```

**设计理由**：
- **体验一致**：无论覆盖率如何，用户每次都看到相同的结果格式和询问
- **逻辑简单**：不需要根据覆盖率决定输出什么
- **真正的用户自主**：用户自己决定是否需要深入分析，不管覆盖率是多少

## 核心工具清单

### 阶段 0（需求澄清）

| 工具 | 功能 | 服务器 |
|------|------|--------|
| `search_indicators_by_vector` | 向量化语义搜索魔数师指标（迭代循环中多次调用） | theme-vector |
| `AskUserQuestion` | 用户确认交互（收敛后或低置信度时触发） | 内置 |

### 阶段 1（主题决策与指标精筛）

| 工具 | 功能 | 服务器 |
|------|------|--------|
| `aggregate_themes_from_indicators` | 从指标列表聚合候选主题（按频次排序） | theme-ontology |
| `get_theme_filter_indicators` | 获取主题下全量筛选指标（时间+机构） | theme-ontology |
| `get_theme_analysis_indicators` | 获取主题下全量分析指标 | theme-ontology |
| `get_indicator_field_mapping` | 【语义增强】获取指标的字段映射 | theme-ontology |
| `get_table_terms` | 【语义增强】获取表字段的术语描述 | theme-ontology |

### 阶段 2（模板推荐）

| 工具 | 功能 | 服务器 |
|------|------|--------|
| `get_theme_templates_with_coverage` | 获取主题模板 + 覆盖率计算 | theme-ontology |
| `get_template_indicators` | 获取模板包含的所有指标（LLM 分析用） | theme-ontology |

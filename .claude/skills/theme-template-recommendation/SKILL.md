# 魔数师主题和模板推荐

## 概述

本 Skill 基于用户自然语言问题，推荐"魔数师"平台的主题（THEME）和模板（TEMPLATE）。

**核心价值**：
- **需求澄清**：通过向量化语义搜索将用户问题映射到魔数师指标，通过人机协作确认映射结果
- **主题推荐**：推荐合适的业务主题
- **指标推荐**：推荐主题下可勾选的核心指标
- **模板推荐**：推荐可直接使用的透视分析/万能查询模板

---

## 输入参数

**用户问题**（必填）：自然语言表述的数据分析需求

```
用户问题: "我想分析南京分行的小微企业贷款风险"
```

**可选参数**：
- `template_type`: 模板类型过滤（"INSIGHT" / "COMBINEDQUERY" / None 默认全部）
- `top_k_themes`: 返回主题数量（默认 3）
- `top_k_templates`: 每种类型返回模板数量（默认 5）

---

## 核心概念：统一映射到指标

**所有用户词组都映射到魔数师指标**，没有"维度"的概念。

| 用户词组 | 映射到的指标 | 说明 |
|----------|--------------|------|
| "贷款余额" | 借据余额 | 分析度量指标 |
| "风险" | 五级分类 | 分析度量指标 |
| "南京分行" | **二级账务机构名称** | 机构筛选指标 |
| "泰州分行" | **二级账务机构名称** | 机构筛选指标 |
| "2024年" | **统计日期** | 时间筛选指标 |

**关键理解**：
- "南京分行"、"泰州分行"、"扬州分行"等具体分行名称，都映射到同一个指标："二级账务机构名称"
- 机构、时间等筛选条件在魔数师中也是**指标**，不是单独的"维度"
- 所有映射结果都是魔数师指标，只是用途不同（分析度量 vs 筛选条件）

### 阶段 1：指标信息获取

**目标**：基于阶段 0 输出的确认指标，获取完整信息用于后续裁决。

```
┌─────────────────────────────────────────────────────────────┐
│  [1.1] 使用阶段 0 的确认指标                                   │
├─────────────────────────────────────────────────────────────┤
│  输入：confirmed_indicators（来自阶段 0）                     │
│                                                              │
│  对每个指标：                                                 │
│  1. get_indicator_full_path(indicator_id) → 业务路径         │
│     → 从 path_nodes 中提取 THEME 信息                        │
│  2. 【语义增强】获取指标的业务描述：                          │
│     - get_indicator_field_mapping(indicator_id) → 字段映射   │
│     - get_table_terms(schema, table) → 术语描述              │
│     - 过滤：只保留与指标字段名匹配的术语                      │
└─────────────────────────────────────────────────────────────┘
```

**语义增强规则**：
- **优先使用术语描述**（term_description）：术语描述通常比指标描述更详细
- **降级策略**：如果术语无描述，使用指标描述（indicator_desc）；如果都无描述，只显示别名
- **术语过滤**：字段名与指标字段名完全匹配（大小写不敏感）
  - 例如：指标字段 `cfc_loan_bal` 匹配术语字段 `CFC_LOAN_BAL`

**效率优化**：
- 只对最终推荐给用户的指标进行语义增强（通常 5-15 个）
- 同一张表的术语只获取一次（缓存机制）

### 阶段 2：主题聚合与决策

**目标**：从匹配指标中聚合 THEME，决策出推荐主题。

**执行步骤**：

```
┌─────────────────────────────────────────────────────────────┐
│  [2.1] THEME 统计聚合                                        │
├─────────────────────────────────────────────────────────────┤
│  从所有匹配指标的 path_nodes 中提取 THEME 信息：               │
│                                                              │
│  统计逻辑：                                                   │
│  - 统计各 THEME 出现频次                                     │
│  - 按频次排序                                                │
│  - 选择 Top 3 THEME 作为推荐主题                             │
│                                                              │
│  示例结果：                                                   │
│  THEME.DS_SSA_RISK。对公贷款借据: 15 个指标                   │
│  THEME.DS_SSA_RISK。对公贷款客户: 8 个指标                    │
│  THEME.DS_SSA_RISK。贷款质量: 5 个指标                       │
└─────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  [2.2] 为每个推荐主题筛选覆盖用户问题的指标                   │
├─────────────────────────────────────────────────────────────┤
│  对每个推荐主题：                                             │
│  - 筛选属于该主题的匹配指标                                   │
│  - 根据用户问题语义，选择能够覆盖用户问题的指标                │
│  - 指标数量不设上限，取决于用户问题的复杂度                    │
│    - 简单问题：可能只需要 2-3 个指标                          │
│    - 复杂问题：可能需要 10+ 个指标                            │
│  - 输出指标 ID、别名、描述                                    │
│                                                              │
│  关键原则：推荐的指标必须能够完整覆盖用户的语义诉求             │
└─────────────────────────────────────────────────────────────┘
```

**为什么采用主题优先？**：
1. 模板数量太多（75,257 个），直接推荐噪音大
2. 相似模板多（不同用户在同一主题下创建相近模板）
3. 主题是天然的过滤器，缩小搜索空间

### 阶段 3：模板推荐（两阶段：程序化初筛 + LLM 精细化判定）

**目标**：在推荐主题范围内，推荐可直接使用的模板。对于覆盖率 < 80% 的候选模板，提供用户主动触发的 LLM 精细化分析。

**核心原则**：
- **覆盖率 >= 80%**：程序化直接推荐
- **覆盖率 < 80%**：输出为候选模板，由用户决定是否需要 LLM 深度分析
- LLM 分析是用户**主动触发**的动作，与覆盖率阈值完全解耦，体验一致

**执行步骤**：

```
┌─────────────────────────────────────────────────────────────────┐
│  [3.1] 程序化初筛                                               │
├─────────────────────────────────────────────────────────────────┤
│  对每个推荐主题调用:                                             │
│  get_theme_templates_with_coverage(                              │
│    theme_id="...",                                               │
│    matched_indicator_ids=[...],                                  │
│    template_type=None,                                            │
│    top_k=20                                                       │
│  )                                                               │
│                                                                   │
│  覆盖率计算:                                                      │
│  coverage = 匹配指标数 / 用户需要的指标数                          │
│                                                                   │
│  分类输出:                                                       │
│  - coverage >= 80% → ✅ 推荐模板（可直接使用）                    │
│  - coverage < 80% → 📋 候选模板（按覆盖率排序）                   │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────────┐
│  [3.2] 始终询问用户                                              │
├─────────────────────────────────────────────────────────────────┤
│  无论是否有 >= 80% 的模板，都输出:                               │
│  - 推荐模板（覆盖率 >= 80%）                                     │
│  - 候选模板列表（覆盖率 < 80%，按覆盖率排序）                     │
│  - 询问：是否需要 LLM 深度分析候选模板？                         │
│                                                                   │
│  💡 用户主动触发后，进入 LLM 精细化判定                          │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼ (用户主动触发)
┌─────────────────────────────────────────────────────────────────┐
│  [3.3] LLM 精细化判定                                           │
├─────────────────────────────────────────────────────────────────┤
│  对每个候选模板:                                                  │
│  1. 获取模板完整信息（调用 get_template_indicators）             │
│  2. 构建 LLM 分析 prompt                                        │
│  3. LLM 分析并输出结构化判定结果                                 │
│                                                                   │
│  LLM 分析内容:                                                   │
│  - 业务场景匹配度: 模板 vs 用户需求                               │
│  - 数据粒度匹配度: 明细级 vs 汇总级                              │
│  - 最终判定: 推荐/不推荐 + 理由                                   │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

**设计理由**：
- **体验一致**：无论覆盖率如何，用户每次都看到相同的结果格式和询问
- **逻辑简单**：不需要根据覆盖率决定输出什么
- **真正的用户自主**：用户自己决定是否需要深入分析，不管覆盖率是多少

## LLM 精细化分析 Prompt

当用户触发 LLM 分析时，使用以下 prompt 模板进行分析：

```
你是银行数据分析专家。请分析以下候选模板与用户需求的业务匹配度。

【用户需求】
{user_question}

【用户确认的指标】
{for indicator in confirmed_indicators}
- {indicator.alias}
  描述: {indicator.description}
{/for}

【候选模板】
{for template in candidate_templates}
- 模板名称: {template.alias}
- 模板描述: {template.description}
- 覆盖率: {template.coverage_ratio}%
- 包含指标: {template.indicators}
{/for}

【分析要求】
对每个候选模板，分析：
1. 业务场景匹配度：该模板的业务场景是什么？与用户需求是否匹配？
2. 数据粒度匹配度：该模板是明细级还是汇总级？
3. 指标覆盖分析：该模板包含的指标是否能满足用户核心需求？
4. 最终判定：推荐/不推荐，并说明理由

【输出格式】
{
  "template_id": "...",
  "business_match_score": 1-5,
  "granularity_match": "明细级/汇总级/不适用",
  "reasoning": "判定理由",
  "recommendation": "推荐/不推荐",
  "supplementary_indicators": ["补充指标列表"]
}
```

## 与 Smart Query 的区别

| 维度         | Smart Query             | 主题模板推荐               |
| ------------ | ----------------------- | -------------------------- |
| **目标**     | 定位物理表和字段        | 推荐主题和模板             |
| **输出**     | 主表 + 字段映射         | 主题 + 指标 + 模板         |
| **用户场景** | 需要 SQL 或了解数据存储 | 需要在魔数师平台拖拉拽分析 |
| **核心路径** | 指标/场景/术语 → 物理表 | 指标 → 主题 → 模板         |

# 魔数师主题和模板推荐

## 概述

本 Skill 基于用户自然语言问题，推荐"魔数师"平台的主题（THEME）和模板（TEMPLATE）。

**核心价值**：
- **需求澄清**：通过向量化语义搜索将用户问题映射到魔数师指标，通过人机协作确认映射结果
- **主题推荐**：推荐合适的业务主题
- **指标推荐**：推荐主题下可勾选的核心指标
- **模板推荐**：推荐可直接使用的透视分析/万能查询模板

---

## 输入参数

**用户问题**（必填）：自然语言表述的数据分析需求

```
用户问题: "我想分析南京分行的小微企业贷款风险"
```

**可选参数**：
- `top_k_themes`: 返回主题数量（默认 3）
- `top_k_templates`: 每种类型返回模板数量（默认 5）

---

## 核心概念：统一映射到指标

**所有用户词组都映射到魔数师指标**，没有"维度"的概念。

| 用户词组 | 映射到的指标 | 说明 | 是否需要用户确认 |
|----------|--------------|------|----------------|
| "**分行" | **二级账务机构名称** | 机构筛选指标 | ❌ 自动应用 |
| "**年" | **数据日期** | 时间筛选指标 | ❌ 自动应用 |

**关键理解**：
- **筛选值**（机构、时间、地区）：直接映射为筛选指标，**自动应用，无需用户确认**
- **分析概念**：向量搜索后经迭代精炼收敛，用户确认的是**分析意图（关键词）**而非具体指标
- 所有映射结果都是魔数师指标，只是用途不同（分析度量 vs 筛选条件）

## 执行流程

### 阶段 0：需求澄清

**目标**：将用户问题映射到魔数师指标，通过人机协作确认分析意图。

**处理原则**：
- **筛选值**（机构、时间、地区）：直接映射，**自动应用，无需用户确认**
- **分析概念**：向量搜索 → 迭代精炼循环收敛 → 用户确认

#### 0.1 关键词提取

```
┌─────────────────────────────────────────────────────────────┐
│  关键词提取                                                  │
├─────────────────────────────────────────────────────────────┤
│  从用户问题中提取业务相关的关键词词组：                        │
│  - "南京分行"                                                │
│  - "小微企业贷款"                                            │
│  - "风险"                                                    │
└─────────────────────────────────────────────────────────────┘
```

#### 0.2 关键词分类与分类处理

```
┌─────────────────────────────────────────────────────────────┐
│  关键词分类与分类处理（内部执行，不展示给用户）                  │
├─────────────────────────────────────────────────────────────┤
│  1. LLM 语义分类（不输出给用户）：                             │
│     对每个关键词判断类型：                                     │
│     - 筛选值类型：                                            │
│       · 机构名称（分行/支行）："南京分行"、"某某支行"           │
│       · 时间词（年/月/日）："2024年"、"今年"、"上个月"         │
│       · 地区/行政区划："江苏省"、"浙江省"、"南京"              │
│     - 分析概念类型：                                          │
│       · 业务术语："小微企业贷款"、"贷款余额"、"不良率"          │
│       · 分析意图词："风险"、"增长"、"占比"                      │
│                                                              │
│  2. 分类处理（静默执行，不展示过程）：                          │
│     - 筛选值类型 → 直接映射到指标（不搜索，不展示）            │
│     - 分析概念类型 → 调用 search_indicators_by_vector        │
│                     进入 0.3 迭代精炼循环                    │
│                                                              │
│  示例（内部数据结构）：                                        │
│  filter_indicators = [                                       │
│    { indicator_id: "二级账务机构名称", value: "南京分行" }    │
│  ]                                                           │
│                                                              │
│  analysis_search_results = {                                  │
│    "小微企业贷款": [                                         │
│      { indicator_id: "...", alias: "借据余额", score: 0.88 },│
│      { indicator_id: "...", alias: "贷款本金", score: 0.75 } │
│    ],                                                        │
│    "风险": [                                                 │
│      { indicator_id: "...", alias: "操作风险", score: 0.82 } │
│    ]                                                         │
│  }                                                           │
└─────────────────────────────────────────────────────────────┘
```

#### 0.3 迭代精炼循环（自动执行，用户不感知）

```
┌─────────────────────────────────────────────────────────────┐
│ [0.3] 迭代精炼循环（自动执行，最多 3 轮）                     │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  输入：                                                      │
│    - analysis_keywords: 0.2 输出的分析概念关键词列表          │
│    - search_results_v1: 0.2 中第 1 轮向量搜索结果            │
│                                                              │
│  循环逻辑：                                                  │
│    WHILE 未收敛 AND 轮次 <= 3:                               │
│      1. LLM 评估当前搜索结果质量                              │
│      2. 若质量达标 → 收敛出口，退出循环                       │
│      3. 若质量不达标 → 生成修正关键词                         │
│      4. 调用 search_indicators_by_vector（修正后关键词）      │
│      5. 合并新旧搜索结果（去重，保留高分候选）                 │
│      6. 轮次 +1                                              │
│    END WHILE                                                 │
│                                                              │
│  两种出口：                                                  │
│    收敛出口：质量达标 → 直接进入 0.4 用户确认                 │
│    超时出口：3 轮未收敛 → 低置信度处理 → 询问用户换关键词      │
│                                                              │
│  输出：                                                      │
│    - final_search_results: 收敛后的候选指标集合              │
│    - iteration_log: 迭代过程记录（用于探查输出展示）           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**收敛判断标准（满足任一即收敛）**：

| 条件 | 说明 |
|------|------|
| 每个分析概念 Top-1 相似度 > 0.80 | 语义命中精准 |
| 本轮修正关键词与上轮完全相同 | 自然收敛，无新变化 |
| 本轮新增候选指标数 = 0 | 搜索结果已饱和 |

**超时出口处理（低置信度）**：

3 轮结束后仍有关键词 Top-1 相似度 < 0.60，进入低置信度流程：
- 停止迭代，不进入用户确认
- 向用户说明哪个关键词无法匹配
- 提供 2~3 个换词建议
- 等待用户重新描述后，重新进入 0.1

**每轮 LLM 评估 Prompt**：

```
你是银行数据分析专家，正在帮助用户精确定位分析指标。

【用户原始问题】
{user_question}

【当前轮次】第 {round} 轮

【当前关键词及搜索结果】
{for keyword, results in current_search_results}
关键词：「{keyword}」
搜索结果 Top-5：
  {for r in results[:5]}
  - {r.alias}（相似度: {r.score}）描述: {r.description}
  {/for}
{/for}

【评估任务】
1. 覆盖评估：用户意图的哪些部分已被覆盖？哪些缺失？
2. 质量判断：当前结果是否达标？（每个关键词 Top-1 > 0.80）
3. 若不达标，诊断问题：
   - 关键词过于宽泛？（结果太杂）→ 收缩
   - 关键词过于口语化？（结果偏离）→ 替换
   - 关键词缺少限定词？（召回不足）→ 扩展
   - 多个关键词指向同一意图？→ 合并
4. 输出修正关键词

【输出格式（JSON）】
{
  "quality_pass": true/false,
  "coverage_assessment": "已覆盖XX，缺失XX",
  "corrections": [
    {
      "original": "原关键词",
      "action": "扩展/收缩/替换/合并",
      "corrected": ["修正后关键词1", "修正后关键词2"],
      "reason": "修正原因"
    }
  ]
}
```

```
┌─────────────────────────────────────────────────────────────┐
│  基于确认关键词 → 最终指标                                    │

│  用户确认后：                                                 │
│  1. 获取用户确认的精炼关键词                                  │
│  2. 复用 0.3 迭代循环中收敛保存的 final_search_results       │
│  3. 调用 batch_get_indicators_themes 获取指标主题信息        │
│  4. 汇总输出最终指标列表                                      │

│  - refined_keywords: 用户确认的分析关键词及关联指标             │
```

**阶段 0 → 阶段 1 传递内容**：
- `refined_keywords`: 迭代精炼收敛后的分析关键词及关联指标
- `filter_indicators`: 自动映射的筛选指标（无需用户确认）
- `matched_indicators`: 用户确认关键词对应的指标列表
- `primary_theme`: 主推荐主题（用于阶段 2 主题推荐参考）
- `normalized_question`: 用户确认后的问题改写

### 阶段 1：指标信息获取

**目标**：基于阶段 0 输出的确认关键词及关联指标，获取完整信息用于后续裁决。

```
┌─────────────────────────────────────────────────────────────┐
│  [1.1] 使用阶段 0 的确认关键词                                   │
├─────────────────────────────────────────────────────────────┤
│  输入：refined_keywords + matched_indicators（来自阶段 0）    │
│                                                              │
│  对每个指标（从 matched_indicators 获取）：                    │
│  1. get_indicator_full_path(indicator_id) → 业务路径         │
│     → 从 path_nodes 中提取 THEME 信息                        │
│  2. 【语义增强】获取指标的业务描述：                          │
│     - get_indicator_field_mapping(indicator_id) → 字段映射   │
│     - get_table_terms(schema, table) → 术语描述              │
│     - 过滤：只保留与指标字段名匹配的术语                      │
└─────────────────────────────────────────────────────────────┘
```

**语义增强规则**：
- **优先使用术语描述**（term_description）：术语描述通常比指标描述更详细
- **降级策略**：如果术语无描述，使用指标描述（indicator_desc）；如果都无描述，只显示别名
- **术语过滤**：字段名与指标字段名完全匹配（大小写不敏感）
  - 例如：指标字段 `cfc_loan_bal` 匹配术语字段 `CFC_LOAN_BAL`

**效率优化**：
- 只对最终推荐给用户的指标进行语义增强（通常 5-15 个）
- 同一张表的术语只获取一次（缓存机制）

---

### 阶段 2：主题聚合与决策

**目标**：从匹配指标中聚合 THEME，确定推荐主题后补全主题全量指标，再由 LLM 精筛覆盖用户问题的指标。

**执行步骤**：

```
┌─────────────────────────────────────────────────────────────┐
│  [2.1] THEME 统计聚合                                        │
├─────────────────────────────────────────────────────────────┤
│  从所有匹配指标的 path_nodes 中提取 THEME 信息：               │
│                                                              │
│  统计逻辑：                                                   │
│  - 统计各 THEME 出现频次                                     │
│  - 按频次排序                                                │
│  - 选择 Top 3 THEME 作为推荐主题                             │
│                                                              │
│  示例结果：                                                   │
│  THEME.DS_SSA_RISK。对公贷款借据: 15 个指标                   │
│  THEME.DS_SSA_RISK。对公贷款客户: 8 个指标                    │
│  THEME.DS_SSA_RISK。贷款质量: 5 个指标                       │
└─────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  [2.2] ★ 主题全量指标补全（新增）                             │
├─────────────────────────────────────────────────────────────┤
│  对每个推荐主题，调用：                                        │
│  get_theme_all_indicators(theme_id)                          │
│                                                              │
│  获取该主题下的完整指标列表，不依赖向量搜索的召回结果。          │
│  目的：消除向量搜索的语义盲区，确保主题内指标不遗漏。           │
│                                                              │
│  数量分支：                                                   │
│  - 全量指标 <= 50 个 → 直接进入 2.3 LLM 逐条精筛             │
│  - 全量指标 > 50 个  → 进入 2.3 LLM 分组精筛                 │
└─────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  [2.3] LLM 从全量指标中精筛覆盖用户问题的指标                 │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  两种模式，根据 2.2 的数量分支选择：                           │
│                                                              │
│  ┌─ 模式 A：逐条精筛（全量 <= 50 个）─────────────────────┐  │
│  │  将全量指标一次性交给 LLM，完整阅读每个指标后判断保留。  │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                              │
│  ┌─ 模式 B：分组精筛（全量 > 50 个）──────────────────────┐  │
│  │  将全量指标按每组 20~30 个分组。                        │  │
│  │  每组独立调用 LLM，判断每个指标"保留 / 排除"。          │  │
│  │  汇总所有组的保留结果 → 最终指标列表。                  │  │
│  │                                                        │  │
│  │  优势：                                                │  │
│  │  · 不引入向量搜索，不产生新的语义盲区                   │  │
│  │  · 每组列表长度可控，LLM 判断精度不下降                 │  │
│  │  · 完全依赖语义理解，而非向量距离                       │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**LLM 精筛 Prompt（模式 A / 模式 B 每组通用）**：

```
你是银行数据分析专家。

【用户原始问题】
{user_question}

【用户确认的分析关键词】
{refined_keywords}

【待筛选指标列表】
{for indicator in candidate_indicators}
- {indicator.alias}（ID: {indicator.id}）
  描述: {indicator.description}
{/for}

【任务】
从上述指标中，筛选出能够覆盖用户分析需求的指标。

筛选原则：
1. 与用户分析关键词语义相关的指标 → 保留
2. 对用户问题有补充分析价值的指标 → 保留
3. 与用户问题无关的指标 → 排除
4. 指标数量不设上限，以完整覆盖用户需求为准

【输出格式（JSON）】
{
  "selected_indicators": [
    {
      "indicator_id": "...",
      "alias": "指标别名",
      "reason": "保留原因"
    }
  ],
  "excluded_count": 排除的指标数量,
  "coverage_assessment": "本组覆盖了用户需求的哪些方面"
}
```

**为什么需要补全这一步？**

| 对比项 | 补全前（仅向量搜索） | 补全后（主题全量精筛） |
|--------|------------------|-------------------|
| 指标来源 | 向量搜索召回（有限） | 主题全量（完整） |
| 遗漏风险 | 高（受搜索质量限制） | 低（主题内无遗漏） |
| 推荐质量 | 依赖用户表述完整度 | 依赖主题定义完整度 |
| 适用场景 | 用户表述清晰时效果好 | 用户表述模糊时兜底 |

### 阶段 3：模板推荐（两阶段：程序化初筛 + LLM 精细化判定）

**目标**：在推荐主题范围内，推荐可直接使用的模板。对于覆盖率 < 80% 的候选模板，提供用户主动触发的 LLM 精细化分析。

**核心原则**：
- **覆盖率 >= 80%**：程序化直接推荐
- **覆盖率 < 80%**：输出为候选模板，由用户决定是否需要 LLM 深度分析
- LLM 分析是用户**主动触发**的动作，与覆盖率阈值完全解耦，体验一致

**执行步骤**：

```
┌─────────────────────────────────────────────────────────────────┐
│  [3.1] 程序化初筛                                               │
├─────────────────────────────────────────────────────────────────┤
│  对每个推荐主题调用:                                             │
│  get_theme_templates_with_coverage(                              │
│    theme_id="...",                                               │
│    matched_indicator_ids=[从 refined_keywords 提取的指标ID列表],   │
│    template_type=None,                                            │
│    top_k=20                                                       │
│  )                                                               │
│                                                                   │
│  覆盖率计算:                                                      │
│  coverage = 匹配指标数 / 用户需要的指标数（来自 refined_keywords）  │
│                                                                   │
│  分类输出:                                                       │
│  - coverage >= 80% → ✅ 推荐模板（可直接使用）                    │
│  - coverage < 80% → 📋 候选模板（按覆盖率排序）                   │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────────┐
│  [3.2] 始终询问用户                                              │
├─────────────────────────────────────────────────────────────────┤
│  无论是否有 >= 80% 的模板，都输出:                               │
│  - 推荐模板（覆盖率 >= 80%）                                     │
│  - 候选模板列表（覆盖率 < 80%，按覆盖率排序）                     │
│  - 询问：是否需要 LLM 深度分析候选模板？                         │
│                                                                   │
│  💡 用户主动触发后，进入 LLM 精细化判定                          │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼ (用户主动触发)
┌─────────────────────────────────────────────────────────────────┐
│  [3.3] LLM 精细化判定                                           │
├─────────────────────────────────────────────────────────────────┤
│  对每个候选模板:                                                  │
│  1. 获取模板完整信息（调用 get_template_indicators）             │
│  2. 构建 LLM 分析 prompt                                        │
│  3. LLM 分析并输出结构化判定结果                                 │
│                                                                   │
│  LLM 分析内容:                                                   │
│  - 业务场景匹配度: 模板 vs 用户需求                               │
│  - 数据粒度匹配度: 明细级 vs 汇总级                              │
│  - 最终判定: 推荐/不推荐 + 理由                                   │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

**设计理由**：
- **体验一致**：无论覆盖率如何，用户每次都看到相同的结果格式和询问
- **逻辑简单**：不需要根据覆盖率决定输出什么
- **真正的用户自主**：用户自己决定是否需要深入分析，不管覆盖率是多少

## LLM 精细化分析 Prompt

当用户触发 LLM 分析时，使用以下 prompt 模板进行分析：

```
你是银行数据分析专家。请分析以下候选模板与用户需求的业务匹配度。

【用户需求】
{user_question}

【用户确认的关键词及关联指标】
{for keyword in refined_keywords}
- {keyword.keyword}
  关联指标: {keyword.matched_indicators}
{/for}

【候选模板】
{for template in candidate_templates}
- 模板名称: {template.alias}
- 模板描述: {template.description}
- 覆盖率: {template.coverage_ratio}%
- 包含指标: {template.indicators}
{/for}

【分析要求】
对每个候选模板，分析：
1. 业务场景匹配度：该模板的业务场景是什么？与用户需求是否匹配？
2. 数据粒度匹配度：该模板是明细级还是汇总级？
3. 指标覆盖分析：该模板包含的指标是否能满足用户核心需求？
4. 最终判定：推荐/不推荐，并说明理由

【输出格式】
{
  "template_id": "...",
  "business_match_score": 1-5,
  "granularity_match": "明细级/汇总级/不适用",
  "reasoning": "判定理由",
  "recommendation": "推荐/不推荐",
  "supplementary_indicators": ["补充指标列表"]
}
```

## 核心工具清单

### 阶段 0（需求澄清）- 向量化搜索

| 工具 | 功能 | 服务器 |
|------|------|--------|
| `search_indicators_by_vector` | **向量化语义搜索魔数师指标**（迭代循环中多次调用） | theme-vector |
| `AskUserQuestion` | 用户确认交互（仅在收敛后或低置信度时触发） | 内置 |
| `batch_get_indicators_themes` | 批量获取指标主题（用于聚合 THEME） | theme-ontology |

### 阶段 1（指标信息获取）- Neo4j

| 工具 | 功能 |
|------|------|
| `get_indicator_full_path` | 获取指标完整业务路径（含 THEME） |
| `get_indicator_field_mapping` | **【语义增强】获取指标的字段映射** |
| `get_table_terms` | **【语义增强】获取表字段的术语描述** |

### 阶段 2-3（主题与模板推荐）- Neo4j

| 工具 | 功能 |
|------|------|
| `batch_get_indicators_themes` | 批量提取指标的 THEME 信息 |
| `get_theme_templates_with_coverage` | 获取主题模板 + 覆盖率计算 |
| `get_template_indicators` | 获取模板包含的所有指标（LLM 分析用） |

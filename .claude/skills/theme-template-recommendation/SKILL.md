# 魔数师主题和模板推荐

## 概述

本 Skill 基于用户自然语言问题，推荐"魔数师"平台的主题（THEME）和模板（TEMPLATE）。

**核心价值**：
- **需求澄清**：通过向量化语义搜索将用户问题映射到魔数师指标，通过人机协作确认映射结果
- **主题推荐**：推荐合适的业务主题
- **指标推荐**：推荐主题下可勾选的核心指标
- **模板推荐**：推荐可直接使用的透视分析/万能查询模板

---

## 输入参数

**用户问题**（必填）：自然语言表述的数据分析需求

```
用户问题: "我想分析南京分行的小微企业贷款风险"
```

**可选参数**：
- `top_k_themes`: 返回主题数量（默认 3）
- `top_k_templates`: 每种类型返回模板数量（默认 5）

---

## 核心概念：统一映射到指标

**所有用户词组都映射到魔数师指标**，没有"维度"的概念。

| 用户词组 | 映射到的指标 | 说明 | 是否需要用户确认 |
|----------|--------------|------|----------------|
| "**分行" | **二级账务机构名称** | 机构筛选指标 | ❌ 自动应用 |
| "**年" | **数据日期** | 时间筛选指标 | ❌ 自动应用 |

**关键理解**：
- **筛选值**（机构、时间、地区）：直接映射为筛选指标，**自动应用，无需用户确认**
- **分析概念**：向量搜索后经迭代精炼收敛，收敛后的搜索词即为分析维度，用户确认的是**分析意图**而非具体指标
- 所有映射结果都是魔数师指标，只是用途不同（分析度量 vs 筛选条件）

## 执行流程

### 阶段 0：需求澄清

**目标**：将用户问题映射到魔数师指标，通过人机协作确认分析意图。

**处理原则**：
- **筛选值**（机构、时间、地区）：直接映射，**自动应用，无需用户确认**
- **分析概念**：向量搜索 → 迭代精炼循环收敛 → 收敛后搜索词直接作为分析维度 → 用户确认

#### 0.1 原始词组提取

从用户问题中提取业务相关的词组，不做任何分类或加工：

```
提取示例：
- "南京分行"
- "小微企业贷款"
- "风险"
```

#### 0.2 词组分类与分类处理

**内部执行，不展示给用户。**

1. **LLM 语义分类**，对每个词组判断类型：
   - **筛选值类型**：
     - 机构名称（分行/支行）："南京分行"、"某某支行"
     - 时间词（年/月/日）："2024年"、"今年"、"上个月"
     - 地区/行政区划："江苏省"、"浙江省"、"南京"
   - **分析概念类型**：
     - 业务术语："小微企业贷款"、"贷款余额"、"不良率"
     - 分析意图词："风险"、"增长"、"占比"

2. **分类处理**（静默执行）：
   - 筛选值类型 → 直接映射到指标，存入 `filter_indicators`
   - 分析概念类型 → 调用 `search_indicators_by_vector`，进入 0.3 迭代精炼循环

**内部数据结构示例**：

```
filter_indicators = [
  { indicator_id: "二级账务机构名称", value: "南京分行" }
]

search_results_v1 = {
  "小微企业贷款": [
    { indicator_id: "...", alias: "借据余额", score: 0.88 },
    { indicator_id: "...", alias: "贷款本金", score: 0.75 }
  ],
  "风险": [
    { indicator_id: "...", alias: "操作风险", score: 0.82 }
  ]
}
```

#### 0.3 迭代精炼循环（自动执行，用户不感知）

**目标**：通过多轮搜索 + LLM 评估，将分析概念收敛为高质量的候选指标集合。**收敛后的搜索词直接作为分析维度**，无需额外归纳步骤。

**循环逻辑**：

```
输入：
  - user_question: 用户原始问题
  - original_concepts: 原始分析概念（锚点，不随迭代变化）
  - analysis_concepts: 当前待搜索的分析概念词组列表
  - converged_concepts: 已收敛概念集合（初始为空）
  - unconverged_concepts: 未收敛概念集合（初始为 analysis_concepts）

WHILE 轮次 < 5:
  轮次 +1
  1. 对 unconverged_concepts 中的每个概念调用 search_indicators_by_vector
  2. 收敛判定：Top-1 相似度 >= 0.80 的概念移入 converged_concepts
  3. 若 unconverged_concepts 为空 → 退出循环（正常收敛出口）
  4. 若轮次已达 5 → 退出循环（超时出口）
  5. 对 unconverged_concepts 调用 LLM 生成下一轮修正搜索词
  6. 若修正词与当前搜索词完全相同 → 退出循环（自然收敛）
  7. 用修正词替换 unconverged_concepts，进入下一轮
END WHILE

输出：
  - converged_search_terms: 收敛后的搜索词列表（即分析维度）
  - final_search_results: 每个搜索词对应的候选指标集合
  - low_confidence_concepts: 未收敛但已尽力收敛的概念
```

**收敛判断标准**：

| 条件 | 说明 | 判断时机 |
|------|------|----------|
| 该概念 Top-1 相似度 >= 0.80 | 语义命中精准 | 每轮搜索后，逐概念判断 |
| 本轮修正词与当前搜索词完全相同 | 自然收敛 | LLM 生成修正词后逐概念判断 |

**超时出口处理（低置信度）**：

5 轮结束后 unconverged_concepts 仍有未收敛概念，进入低置信度流程：

- **低置信度展示**：展示已收敛维度（默认勾选）和未收敛维度（带警告、默认不勾选），由用户决定是否纳入未收敛维度继续执行，或重新描述换词

**每轮 LLM 修正搜索词生成 Prompt**（仅对未收敛概念执行）：

```
你是银行数据分析专家，正在帮助用户精确定位分析指标。

【用户原始问题】
{user_question}

【当前轮次】第 {round} 轮（共最多 {max_rounds} 轮）

【未收敛概念及其搜索结果】
{pending_search_results_str}

【已收敛概念（仅供参照，无需处理）】
{converged_concepts_str}

【意图锚定约束】（重要！）
1. **修正词必须保留原始语义单元的完整业务含义**
   - 专有业务术语不可拆分（如"结算客户"是一个独立语义单元，不能拆成"结算"+"客户"）
   - 修正后的搜索词必须覆盖原始概念的核心语义
2. **禁止引入新语义域**
   - 不能用含义完全不同的词替换（如"存款"→"贷款"是偏离）
   - 只能在同一语义域内优化表达方式
3. **偏离度检测**
   - 如果修正词与原始概念的语义偏离度 > 50%，触发 DEVIATION_ABORT
   - 此时应保留原始概念，不做任何修正

【诊断任务】
基于已收敛的概念，针对未收敛概念及其搜索结果生成下一轮更精准的搜索词。

【诊断方向引导】
1. 搜索词过于口语化 → 替换为银行业专业术语
2. 搜索词含义模糊 → 拆分为更明确的表达（最多拆分为2个搜索词）
3. 多个概念指向同一意图 → 合并为一个更精准的词
4. 搜索词偏离原始语义 → 保留原始词或做微小调整
5. 搜索词包含统计意图（如"数/量/合计/总额/占比/率"等）→ 剥离统计意图，只保留核心业务实体词
   - 原因：Smart BI 平台具备自动聚合统计能力，指标库中存储的是业务实体属性，而非预聚合结果
   - 示例："结算客户数" → "结算客户"；"贷款余额合计" → "贷款余额"；"不良贷款率" → "不良贷款"（率是计算结果）
   - 注意：如果原词本身就是一个标准银行术语（如"不良率"本身即为行业通用指标名），则保留不拆

【约束条件】
- 输出的新搜索词数量应尽量与未收敛概念数量保持一致或更少（合并时可减少），不允许无限扩展
- 不要输出已收敛概念作为新搜索词
- 使用银行业标准术语，不使用口语化表达
- 如果无法改进，输出原始概念作为新搜索词

【输出格式】
直接返回 JSON，格式为：{{"new_concepts": ["新搜索词1", "新搜索词2"], "reasoning": "诊断说明"}}
不要使用任何 markdown 代码块包裹。
```

#### 0.4 用户确认分析维度 + 勾选引导

基于 0.3 所有的分析概念（收敛的和未收敛的分析概念）及其关联候选指标，展示给用户确认。

**关键说明**：
- **分析维度 = 0.3 收敛后的搜索词**，无需额外归纳，直接展示
- 指标仅作为辅助说明，帮助用户理解该维度的含义
- 筛选值不展示在选项中，仅在说明中标注"自动应用"

**⚠️ 勾选引导（核心改进）**：

多个分析维度同时勾选时，如果维度间导向的主题方向完全不同，会导致系统同时推荐两组互不相干的主题，让用户难以聚焦。

**核心改进**：通过查询 Neo4j 获取每个维度的真实 theme_id，基于**相似度加权的 Jaccard 相似度**判断维度间主题交叉程度，避免仅凭指标别名猜测主题导致的误判。

**处理流程（3 步）**：

1. **查 Neo4j（批量，一次请求）**：
   - 对每个分析维度，取其 Top-20 候选指标（来自 0.3 搜索结果，含相似度分数）
   - 合并去重后，一次调用 `batch_get_indicator_themes(indicator_ids)` 获取所有指标的主题归属
   - 按维度分组聚合：每个维度命中的所有 theme 及其**对该维度的贡献权重**（取指标最大相似度）

2. **算加权 Jaccard**：计算维度两两之间的加权 Jaccard 相似度
   - 传统 Jaccard：`J = |交集| / |并集|`（所有 theme 等权重）
   - **加权 Jaccard**：`J = Σ(min(w_A, w_B)) / Σ(max(w_A, w_B))`
     - `w_A`：theme 对维度 A 的贡献权重（= 映射到该 theme 的指标相似度之和）
     - `w_B`：theme 对维度 B 的贡献权重
   - Jaccard >= 0.5 → 主题高度重叠，两个维度互补性强（可同时勾选）
   - Jaccard < 0.5  → 两个维度导向的主题方向完全不同，同时勾选会让推荐结果分散（建议优先勾选用户最关心的核心维度）

**加权 Jaccard 计算示例**：

| Theme | 维度A权重(w_A) | 维度B权重(w_B) | min | max |
|-------|----------------|----------------|-----|-----|
| 个人存款 | 0.92 | 0.85 | 0.85 | 0.92 |
| 小微贷款 | 0.88 | 0.38 | 0.38 | 0.88 |
| 企业贷款 | 0.31 | 0.92 | 0.31 | 0.92 |
| **合计** | - | - | **1.54** | **2.72** |

```
加权 Jaccard = 1.54 / 2.72 = 0.57  (>= 0.5，两个维度互补)
```

**⚠️ 权重衰减规则**：
- 相似度 >= 0.80 → 全权重（1.0）
- 相似度 >= 0.50 → 半权重（0.5）
- 相似度 < 0.50 → 极低权重（0.1）或排除

3. **LLM 引导**：将 theme 信息 + 加权 Jaccard 矩阵注入 Prompt，LLM 综合分析后生成引导建议

**引导输出结构**：

```
dimension_guidance: {
  "has_conflict": true/false,          // 【程序化】是否存在主题交叉（has_conflict = not can_select_all）
  "can_select_all": true/false,        // 【程序化】所有维度是否可以全部勾选（所有两两加权 Jaccard >= 阈值）
  "recommended_first": ["维度A"],       // 【LLM】建议优先勾选的核心维度
  "conflict_analysis": "维度A主要导向XX主题方向，维度B主要导向YY主题方向，两者推荐的主题几乎没有重叠。同时勾选会让推荐结果分散，建议优先选择核心维度。",  // 【LLM】业务语言描述
  "selection_advice": "各维度导向的主题方向不同，建议优先勾选您最关心的核心维度以获得聚焦的推荐。",  // 【程序化】与 can_select_all 严格对应
  "dimension_analysis": [
    {
      "dimension": "维度名",
      "matched_themes": [
        {"theme": "主题1", "weight": 0.92},
        {"theme": "主题2", "weight": 0.88}
      ],                                   // 命中的主题及贡献权重
      "theme_count": 4,                    // 命中主题数量
      "total_weight": 1.80,               // 权重总和（用于排序）
      "independence_score": 0.0-1.0,       // 独立性得分（LLM 给出）
      "core_concept_score": 0.0-1.0,       // 核心概念得分（LLM 给出）
      "recommendation": "优先/可选/建议后选"
    }
  ]
}
```

**异常处理**：若 LLM 调用失败（网络/解析错误），`dimension_guidance` 为 `None`，跳过引导步骤继续执行后续流程。

**AskUserQuestion 配置**：

## 核心工具清单

### 阶段 0（需求澄清）

| 工具 | 功能 | 服务器 |

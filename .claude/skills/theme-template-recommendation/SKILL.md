# 魔数师主题和模板推荐

## 概述

本 Skill 基于用户自然语言问题，推荐"魔数师"平台的主题（THEME）和模板（TEMPLATE）。

**核心价值**：
- **需求澄清**：通过向量化语义搜索将用户问题映射到魔数师指标，通过人机协作确认映射结果
- **主题推荐**：推荐合适的业务主题
- **指标推荐**：推荐主题下可勾选的核心指标
- **模板推荐**：推荐可直接使用的透视分析/万能查询模板

---

## 输入参数

**用户问题**（必填）：自然语言表述的数据分析需求

```
用户问题: "我想分析南京分行的小微企业贷款风险"
```

**可选参数**：
- `top_k_themes`: 返回主题数量（默认 3）
- `top_k_templates`: 每种类型返回模板数量（默认 5）

---

## 核心概念：统一映射到指标

**所有用户词组都映射到魔数师指标**，没有"维度"的概念。

| 用户词组 | 映射到的指标 | 说明 | 是否需要用户确认 |
|----------|--------------|------|----------------|
| "**分行" | **二级账务机构名称** | 机构筛选指标 | ❌ 自动应用 |
| "**年" | **数据日期** | 时间筛选指标 | ❌ 自动应用 |

**关键理解**：
- **筛选值**（机构、时间、地区）：直接映射为筛选指标，**自动应用，无需用户确认**
- **分析概念**：向量搜索后经迭代精炼收敛，收敛后的搜索词即为分析维度，用户确认的是**分析意图**而非具体指标
- 所有映射结果都是魔数师指标，只是用途不同（分析度量 vs 筛选条件）

## 执行流程

### 阶段 0：需求澄清

**目标**：将用户问题映射到魔数师指标，通过人机协作确认分析意图。

**处理原则**：
- **筛选值**（机构、时间、地区）：直接映射，**自动应用，无需用户确认**
- **分析概念**：向量搜索 → 迭代精炼循环收敛 → 收敛后搜索词直接作为分析维度 → 用户确认

#### 0.1 原始词组提取

从用户问题中提取业务相关的词组，不做任何分类或加工：

```
提取示例：
- "南京分行"
- "小微企业贷款"
- "风险"
```

#### 0.2 词组分类与分类处理

**内部执行，不展示给用户。**

1. **LLM 语义分类**，对每个词组判断类型：
   - **筛选值类型**：
     - 机构名称（分行/支行）："南京分行"、"某某支行"
     - 时间词（年/月/日）："2024年"、"今年"、"上个月"
     - 地区/行政区划："江苏省"、"浙江省"、"南京"
   - **分析概念类型**：
     - 业务术语："小微企业贷款"、"贷款余额"、"不良率"
     - 分析意图词："风险"、"增长"、"占比"

2. **分类处理**（静默执行）：
   - 筛选值类型 → 直接映射到指标，存入 `filter_indicators`
   - 分析概念类型 → 调用 `search_indicators_by_vector`，进入 0.3 迭代精炼循环

**内部数据结构示例**：

```
filter_indicators = [
  { indicator_id: "二级账务机构名称", value: "南京分行" }
]

search_results_v1 = {
  "小微企业贷款": [
    { indicator_id: "...", alias: "借据余额", score: 0.88 },
    { indicator_id: "...", alias: "贷款本金", score: 0.75 }
  ],
  "风险": [
    { indicator_id: "...", alias: "操作风险", score: 0.82 }
  ]
}
```

#### 0.3 迭代精炼循环（自动执行，用户不感知）

**目标**：通过多轮搜索 + LLM 评估，将分析概念收敛为高质量的候选指标集合。收敛后的搜索词**直接作为分析维度**，无需额外归纳步骤。

**循环逻辑**：

```
输入：
  - analysis_concepts: 0.2 输出的分析概念词组列表
  - search_results_v1: 0.2 中第 1 轮向量搜索结果
  - converged_concepts: 已收敛概念集合（初始为空）
  - unconverged_concepts: 未收敛概念集合（初始为 analysis_concepts）

WHILE 轮次 < 5:
  轮次 +1
  1. 对 unconverged_concepts 中的每个概念调用 search_indicators_by_vector
  2. 收敛判定：Top-1 >= 0.80 的概念移入 converged_concepts，从 unconverged_concepts 移除
  3. 若 unconverged_concepts 为空 → 退出循环（正常收敛出口）
  4. 若轮次已达 5 → 退出循环（超时出口，进入低置信度流程）
  5. 对 unconverged_concepts 调用 LLM 生成下一轮修正搜索词
  6. 若修正词与当前搜索词完全相同 → 退出循环（自然收敛，无新变化）
  7. 用修正词替换 unconverged_concepts 的 key，进入下一轮
END WHILE

输出：
  - converged_search_terms: 收敛后的搜索词列表（即分析维度，来自 converged_concepts）
  - final_search_results: 每个搜索词对应的候选指标集合（仅保留收敛概念的最终结果）
  - iteration_log: 迭代过程记录（用于探查输出展示）
```

**收敛判断标准（满足任一即收敛）**：

| 条件 | 说明 | 判断时机 |
|------|------|----------|
| 该概念 Top-1 相似度 >= 0.80 | 语义命中精准 | 每轮搜索后，逐概念判断，满足则移入已收敛集合 |
| 所有概念均已收敛 | 未收敛集合为空 | 每轮收敛判定后，整体判断，满足则退出循环 |
| 本轮修正词与当前搜索词完全相同 | LLM 无法进一步优化 | LLM 生成修正词后判断，满足则退出循环 |

**超时出口处理（低置信度）**：

5 轮结束后 unconverged_concepts 仍不为空（即存在 Top-1 相似度 < 0.80 的概念），进入低置信度流程：
- 停止迭代，进入低置信度确认交互，展示已收敛维度（默认勾选）和未收敛维度（带警告、默认不勾选），由用户决定是否纳入未收敛维度继续执行，或重新描述换词

**每轮 LLM 修正搜索词生成 Prompt**（仅对未收敛概念执行）：

```
你是银行数据分析专家，正在帮助用户精确定位分析指标。

【用户原始问题】
{user_question}

【当前轮次】第 {round} 轮

【未收敛分析概念及搜索结果】
{for concept, results in unconverged_concepts}
分析概念：「{concept}」
搜索结果 Top-5：
  {for r in results[:5]}
  - {r.alias}（相似度: {r.score}）描述: {r.description}
  {/for}
{/for}

【诊断任务】
针对每个未收敛的分析概念（Top-1 相似度 < 0.80），诊断问题并生成修正搜索词：
- 搜索词过于宽泛？（结果太杂）→ 收缩
- 搜索词过于口语化？（结果偏离）→ 替换
- 搜索词缺少限定词？（召回不足）→ 扩展
- 多个搜索词指向同一意图？→ 合并

【输出格式（JSON）】
{
  "corrections": [
    {
      "original": "原搜索词",
      "action": "扩展/收缩/替换/合并",
      "corrected": "修正后搜索词",
      "reason": "修正原因"
    }
  ],
  "normalized_question": "规范化后的问题描述（若本轮后预计全部收敛则输出）"
}
```

#### 0.4 用户确认分析维度 + 勾选引导

基于 0.3 所有的分析概念（收敛的和未收敛的分析概念）及其关联候选指标，展示给用户确认。

**关键说明**：
- **分析维度 = 0.3 收敛后的搜索词**，无需额外归纳，直接展示
- 指标仅作为辅助说明，帮助用户理解该维度的含义
- 筛选值不展示在选项中，仅在说明中标注"自动应用"

**⚠️ 勾选引导（核心改进）**：

多个分析维度同时勾选时，可能出现"主题交叉干扰"问题，导致推荐效果下降。

**核心改进**：通过查询 Neo4j 获取每个维度的真实 theme_id，基于 theme_id 集合计算 Jaccard 相似度，
避免仅凭指标别名猜测主题导致的误判。

**处理流程（3 步）**：

1. **查 Neo4j**：对每个维度的全部指标（Top-20），调用 `batch_get_indicator_themes()` 批量获取其所属的 THEME 节点
2. **算 Jaccard**：计算维度两两之间的 Jaccard 相似度 = |交集 theme| / |并集 theme|
   - Jaccard >= 0.5 → 主题高度重叠，两个维度互补性强（可同时勾选）
   - Jaccard < 0.5  → 主题几乎无交集，存在主题交叉干扰（建议优先勾选核心维度）
3. **LLM 引导**：将 theme 信息 + Jaccard 矩阵注入 Prompt，LLM 综合分析后生成引导建议

**引导输出结构**：

```
dimension_guidance: {
  "has_conflict": true/false,          // 是否存在主题交叉（基于 Jaccard < 0.5 判定）
  "recommended_first": ["维度A"],       // 建议优先勾选的核心维度
  "conflict_analysis": "维度A命中了X个主题，维度B命中了Y个主题，Jaccard=xx（<0.5=有冲突），...",
  "selection_advice": "勾选建议（1-2句话）",
  "dimension_analysis": [
    {
      "dimension": "维度名",
      "matched_themes": ["主题1", "主题2"],  // 命中的真实主题别名列表
      "theme_count": 4,                      // 命中主题数量
      "independence_score": 0.0-1.0,         // 独立性得分（LLM 给出）
      "core_concept_score": 0.0-1.0,        // 核心概念得分（LLM 给出）
      "recommendation": "优先/可选/建议后选"
    }
  ]
}
```

**异常处理**：若 LLM 调用失败（网络/解析错误），`dimension_guidance` 为 `None`，跳过引导步骤继续执行后续流程。

**AskUserQuestion 配置**：

> **注意**：当 `dimension_guidance.has_conflict = true` 时，应在问题描述中增加引导提示，
> 告知用户"建议优先勾选最核心的维度，全选不一定最优"；若 `has_conflict = false`，
> 说明各维度主题一致或互补，无需额外引导。

```
AskUserQuestion:
  questions:
    - question: "以下筛选条件已自动识别，将直接应用："
      header: "筛选条件（自动应用）"
      options:
        - label: "🏦 二级账务机构名称 = '南京分行'"
          value: "FILTER:auto:二级账务机构名称:南京分行"
          description: "自动映射，无需确认"

    - question: "基于您的分析需求，精炼出以下分析维度，请确认："
      header: "分析维度（请确认）"
      multiSelect: true
      options:
        - label: "📊 小微企业贷款风险"
          value: "DIMENSION:小微企业贷款风险"
          description: "关联指标：借据余额、贷款本金、五级分类、不良率、逾期率"

    - question: "请确认或修改分析需求描述："
      header: "需求确认"
      options:
        - label: "\"分析南京分行的小微企业贷款信用风险\"（推荐）"

```

#### 0.5 整理输出，传递给阶段 1

用户确认后，整理结构化输出：

```
analysis_dimensions:  用户确认的分析维度列表
                      （= 0.3 收敛后搜索词 + 每个词对应的候选指标）
filter_indicators:    自动映射的筛选指标（来自 0.2）
matched_indicators:   所有分析维度下关联指标的合并去重列表
normalized_question:  规范化问题描述（来自 0.3 收敛时输出）
```

### 阶段 1：主题定位与指标裁决

**目标**：从阶段 0 获取确认的分析维度及关联指标，聚合推荐主题并精筛覆盖用户需求的指标。

#### 1.1 THEME 统计聚合

从所有匹配指标中聚合 THEME 信息：
1. 调用 `aggregate_themes_from_indicators(matched_indicators)` 批量获取指标的主题信息
2. 统计各 THEME 出现频次并排序
3. 选择 Top 3 THEME 作为初始推荐主题

**输出示例**：
```json
{
  "candidate_themes": [
    {
      "theme_id": "THEME.xxx",
      "theme_alias": "个人存款",
      "theme_level": 4,
      "theme_path": "自主分析 > 负债板块 > 个人存款 > 个人存款",
      "frequency": 5,
      "matched_indicator_ids": ["INDICATOR.xxx", ...]
    }
  ]
}
```

> **注意**：`aggregate_themes_from_indicators` 返回结果中已包含 `theme_path` 字段，
> 该字段为从"自主分析"到该主题的完整路径，方便用户在魔数师平台中快速定位主题。

#### 1.2 指标补全

对每个推荐主题执行指标补全操作：

**步骤 1.2.1**：调用 `get_theme_filter_indicators(theme_id)` 获取主题下的全量筛选指标

**筛选指标识别规则**（在 MCP 工具内部实现）：
- **时间筛选指标**：指标别名包含 `数据日期` 或 `ETL数据日期`
- **机构筛选指标**：指标别名包含 `机构名称`、`机构编号`、`管理机构名称`、`管理机构编号`、`账务机构名称`、`账务机构编号`

**步骤 1.2.2**：调用 `get_theme_analysis_indicators(theme_id)` 获取主题下的全量分析指标

分析指标为主题下除去筛选指标外的所有指标。







#### 1.3 LLM 裁决

**目标**：基于 1.2 补全的全量指标（筛选指标 + 分析指标），由 LLM 分两步完成裁决：
1. **第一步**：判断当前主题是否能够支撑用户需求
2. **第二步**：在能够支撑的前提下，精筛出覆盖用户需求的具体指标

**两步裁决均在同一次 LLM 调用中完成**，通过输出结构区分。

**裁决 Prompt**：

```
你是银行数据分析专家，需要从主题的全量指标中精准定位能覆盖用户需求的指标。

【用户原始问题】
{user_question}

【用户确认的分析维度】
{for dim in analysis_dimensions}
- 「{dim}」关联指标: {dim.matched_indicators}
{/for}

【当前主题】
主题名称: {theme_name}
主题路径: {theme_path}

【主题下全量筛选指标】
{for ind in filter_indicators}
- {ind.alias}（类型: {ind.type}）
{/for}

【主题下全量分析指标】
{for ind in analysis_indicators}
- {ind.alias}
{/for}

【裁决任务】

第一步：主题可用性判断
判断该主题是否能够支撑用户的分析需求。
判断依据：
- 主题的业务领域是否与用户需求匹配？
- 主题下的分析指标是否覆盖用户的核心分析意图？
- 主题下的筛选指标是否支持用户所需的筛选条件（机构、时间等）？

第二步：指标精筛（仅在主题可用时执行）
从筛选指标和分析指标两部分中，分别定位出能覆盖用户分析需求的指标。

精筛原则：
- 筛选指标：保留与用户筛选条件（机构、时间）匹配的指标
- 分析指标：保留与用户分析维度语义相关、或有补充分析价值的指标
- 排除与用户需求无关的指标
- 指标数量不设上限，以完整覆盖用户需求为准

【输出格式（JSON）】
{
  "theme_id": "...",
  "theme_name": "...",
  "is_supported": true/false,
  "support_reason": "主题可用性判断理由",
  "selected_filter_indicators": [
    {
      "indicator_id": "...",
      "alias": "指标别名",
      "type": "机构筛选指标/时间筛选指标",
      "reason": "选取原因"
    }
  ],
  "selected_analysis_indicators": [
    {
      "indicator_id": "...",
      "alias": "指标别名",
      "reason": "选取原因"
    }
  ],
  "unsupported_dimensions": ["无法覆盖的分析维度（如有）"]
}
```

**裁决结果处理**：

| 裁决结果 | 处理方式 |
|---------|---------|
| `is_supported: true` | 进入阶段 2，以精筛后的指标集合为基准做模板匹配 |
| `is_supported: false` | 降级为备选主题，在输出中标注"不推荐"并说明原因 |
| 所有主题均不支持 | 向用户说明无匹配主题，建议换词重新描述 |

**本阶段最终输出（传递给阶段 2）**：

```
recommended_themes = [
  {
    theme_id: "...",
    theme_name: "...",
    theme_path: "...",
    is_supported: true,
    selected_filter_indicators: [...],   // 精筛后的筛选指标
    selected_analysis_indicators: [...], // 精筛后的分析指标（含语义增强描述）
    unsupported_dimensions: []
  },
  ...

```

### 阶段 2：模板推荐

**目标**：基于阶段 1 精筛后的指标集合，检索匹配度达标的模板，并由 LLM 结合用户问题对每个模板的指标构成进行开卷分析，输出可用性说明与缺口提示。

**整体流程**：

```
2.1 MCP 检索模板（带降级策略）
        ↓
   有达标模板（>= 80%）？
        ├─ 是 → 返回达标模板列表
        └─ 否 → 降级推荐：覆盖率最高 + 热度最高的模板（最多 2 个）
        ↓
2.2 LLM 可用性与缺口分析
        ↓
   结合用户问题，分析每个模板的指标构成，输出可用性与缺口说明
```

#### 2.1 MCP 检索模板

对每个 `is_supported: true` 的推荐主题，调用：

```
get_theme_templates_with_coverage(
  theme_id="...",
  matched_indicator_aliases=[
    ...selected_filter_indicators 的 alias,
    ...selected_analysis_indicators 的 alias
  ],
  top_k=10
)
```

**工具过滤逻辑**：

1. **第一步**：过滤掉 `heat = 0` 的模板（无使用记录的模板不参与推荐）
2. **第二步**：在 `heat > 0` 的模板中查找覆盖率 >= 80% 的达标模板
3. **第三步**（降级策略）：如果没有 >= 80% 的模板，降级推荐：
   - 覆盖率最高的模板（1 个）
   - 热度最高的模板（1 个，如果与覆盖率最高不同）
   - 如果是同一个模板，则只输出 1 个

**工具返回结构**：

```json
{
  "has_qualified_templates": true/false,
  "matched_templates": [...],
  "fallback_reason": "降级原因（仅当 has_qualified_templates = false 时）"
}
```

**返回字段说明**：

| 字段 | 说明 |
|------|------|
| `has_qualified_templates` | 是否有覆盖率 >= 80% 的达标模板 |
| `matched_templates` | 达标模板列表，或降级推荐的模板（最多 2 个） |
| `fallback_reason` | 降级原因说明（仅当无达标模板时返回） |

#### 2.2 LLM 可用性与缺口分析

**目标**：对 2.1 返回的模板，LLM 结合用户问题与模板的完整指标构成，分析该模板的实际可用性，并对缺失指标（`missing_indicator_aliases`）给出具体的影响说明与补充建议。

**执行规则**：
- 若 `has_qualified_templates = false`（降级推荐）：**必须**向用户说明"未找到覆盖率 >= 80% 的达标模板，以下为参考推荐"
- 覆盖率 = 100%（`missing_indicator_aliases` 为空）→ 标记为"完整覆盖"，跳过缺口分析
- 覆盖率 < 100%（`missing_indicator_aliases` 不为空）→ 执行缺口分析

**LLM 可用性与缺口分析 Prompt**：

```
你是银行数据分析专家。请基于用户问题，对以下模板的指标构成进行分析，
判断该模板的实际可用性，并说明缺失指标的影响与补充建议。

【用户原始问题】
{user_question}

【用户确认的分析维度及关联指标】
{for dim in analysis_dimensions}
- 「{dim}」关联指标: {dim.matched_indicators}
{/for}

【模板信息】
模板名称: {template.template_alias}
模板描述: {template.template_description}
覆盖率: {template.coverage_ratio * 100}%

【模板包含的全量指标】
{for ind in template.all_template_indicators}
- {ind.alias}：{ind.description}
{/for}

【缺失指标（用户需求中未被该模板覆盖的部分）】
{for ind in missing_indicators}
- {ind.alias}（类型: 筛选指标/分析指标）：{ind.description}
{/for}

【分析任务】
1. 结合模板的全量指标与用户问题，评估该模板能在多大程度上满足用户需求
2. 针对每个缺失指标，判断其对用户分析的重要程度及缺失影响
3. 给出整体可用性结论与补充建议

【输出格式（JSON）】
{
  "template_id": "...",
  "overall_usability": "可直接使用 / 补充后可用 / 缺口较大建议谨慎",
  "usability_summary": "整体可用性的一句话说明，供用户快速判断",
  "missing_indicator_analysis": [
    {
      "indicator_alias": "缺失指标别名",
      "importance": "核心 / 辅助 / 可忽略",
      "impact": "缺失该指标对用户分析的具体影响",
      "supplement_suggestion": "补充建议（无需补充则填'无'）"
    }
  ]
}
```

**可用性等级说明**：

| 可用性等级 | 含义 | 展示标记 |
|-----------|------|---------|
| 可直接使用 | 缺失指标均为辅助或可忽略级别，不影响核心分析 | ✅ |
| 补充后可用 | 缺失部分辅助指标，在模板基础上补充后可满足需求 | 🔧 |
| 缺口较大建议谨慎 | 缺失核心指标，需较多调整才能满足需求 | ⚠️ |

## 核心工具清单

### 阶段 0（需求澄清）

| 工具 | 功能 | 服务器 |
|------|------|--------|
| `search_indicators_by_vector` | 向量化语义搜索魔数师指标（迭代循环中多次调用） | theme-vector |
| `AskUserQuestion` | 用户确认交互（收敛后或低置信度时触发） | 内置 |

### 阶段 1（主题定位与指标裁决）

| 步骤 | 工具 | 功能 |
|------|------|------|
| 1.1 主题统计聚合 | `aggregate_themes_from_indicators` | 从指标列表聚合候选主题（按频次排序） |
| 1.2 全量指标补全 | `get_theme_filter_indicators` | 获取主题下全量筛选指标 |
| 1.2 全量指标补全 | `get_theme_analysis_indicators` | 获取主题下全量分析指标 |

### 阶段 2（模板推荐）

| 工具 | 功能 |
|------|------|
| `get_theme_templates_with_coverage` | 获取主题模板 + 覆盖率计算 |
| `get_template_indicators` | 获取模板包含的所有指标（LLM 分析用） |
